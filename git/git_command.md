# Шпаргалка по Git

*Git — распределённая система контроля версий (Version Control System, VCS). Она помогает отслеживать изменения в коде, возвращаться к старым версиям и работать над проектом вместе с другими разработчиками.*

**Git и CALMS**

Если вспомнить модель CALMS (Culture, Automation, Lean, Measurement, Sharing), то Git затрагивает все её компоненты:

- Culture — культура совместной работы и прозрачности.
- Automation — Git является точкой входа для CI/CD, инфраструктуры как кода, автоматизации.
- Lean — история коммитов помогает дробить изменения, работать с минимальными инкрементами.
- Measurement — из Git легко извлекаются метрики эффективности DevOps-изменений (частота деплоев, время восстановления после инцидента и т. д.).
- Sharing — ветки, пулл-реквесты, ревью — всё это обмен знаниями.

## Распределённая модель: у каждого — вся история

Первая важная особенность Git — его распределённость. То есть у каждого разработчика на локальной машине хранится вся история изменений — не только текущие файлы, но и все коммиты и ветки, даже теги. В отличие от SVN или других централизованных систем, вам не нужно обращаться к серверу, чтобы просмотреть историю или сделать коммит. Вы можете работать офлайн, коммитить локально, делать ветки и даже экспериментировать со слияниями — всё без подключения к интернету.

Почему это важно:

- работа становится быстрой и надёжной: никакой зависимости от центрального сервера;
- можно создавать локальные эксперименты без риска сломать что-то у всех;
- Git идеально подходит для CI/CD, где каждый пайплайн может клонировать код и иметь полный контекст.

Итак, Git — это распределённая система, в которой каждый работает с локальной копией репозитория, а не напрямую с сервером. Изменения можно вносить офлайн и синхронизировать с сервером, когда нужно, — благодаря этому Git остаётся гибким, надёжным и устойчивым к сбоям.

## Модель трёх состояний: рабочая директория, индекс, репозиторий

Вторая важная особенность Git - он отслеживает состояние файлов на трёх уровнях:
1. **Working directory (рабочая директория)** — здесь вы редактируете файлы, создаёте новые, удаляете — это просто обычная файловая система.
2. **Staging area (или Index)** — сюда вы «готовите» изменения для будущего коммита с помощью ```git add```.
3. **Repository (локальный репозиторий)** — это база данных Git, где лежат все сделанные коммиты. Вы пополняете её с помощью ```git commit```.

При этом файлы могут находиться в трёх состояниях:
1. ```modified``` — файл был изменён, но Git пока о нём «не знает». Эти изменения находятся в рабочей директории.
2. ```staged``` — файл был добавлен в индекс, то есть Git готов включить его в следующий снимок (коммит).
3. ```committed``` — файл уже стал частью истории изменений. Его состояние зафиксировано в локальном репозитории.

Каждый коммит имеет уникальный SHA-1 хеш — последовательность из 40 символов, которая создаётся на основе содержимого, метаданных и родительских коммитов. Именно поэтому история Git почти неизменяема: если изменить даже один байт — хеш изменится, и Git это заметит.


**HEAD, ветки и хранилище объектов**
Когда вы переключаетесь между ветками (```git checkout branch```), вы на самом деле просто перемещаете указатель ```HEAD``` — это ссылка на текущий коммит. Ветки — это тоже ссылки (refs), просто с красивыми именами.
Внутри Git устроено так:
- все коммиты — это объекты, хранящиеся в ```.git/objects```;
- ветки — это просто указатели на определённые коммиты;
- ```HEAD``` — текущий указатель на ветку или прямо на коммит.
Если коммиты — это коробки с файлами, то ветки — это бирки, приклеенные к коробкам. Git просто двигает бирки, а история сохраняется.


# Ветвление и стратегии работы с ветками

Ветки позволяют вести работу параллельно и без хаоса: десятки, а то и сотни разработчиков меняют код в одном репозитории и не мешают друг другу. Но чем больше гибкость, тем больше и ответственность: если неправильно выбрать стратегию ветвления — можно легко запутаться, потерять контроль или парализовать весь процесс доставки.

Напомним, что ветка в Git — это указатель на последний коммит в цепочке. Вся история хранится в виде графа: коммиты — это узлы, каждый ссылается на родительский. Git не копирует файлы целиком, он работает со снапшотами изменений. Поэтому создание ветки занимает миллисекунды и не требует дополнительного места. Ветки в Git дешёвые, и это важно.

Допустим, мы создаём ветку `feature/login-form`. Git просто создаёт новый указатель (ref), который ссылается на текущий коммит. С этого момента все наши новые коммиты будут двигать этот указатель вперёд. Основная ветка (`main`, `master`) при этом остаётся на месте, пока мы сами не решим объединить изменения.

**Branch (ветка, бранч)** — это указатель на коммит. Мы можем представить его как ярлычок, приклеенный к определённой точке в истории. Когда мы коммитим, ярлычок сдвигается вперёд. Ветка — это не копия проекта, а просто ссылка.

`HEAD` — это текущая позиция. Он указывает, где мы «находимся» в истории. Обычно он ссылается на ветку, а та — на коммит (но может и напрямую на коммит — detached HEAD).

```bash
git init my-demo-repo ## инициализация нового репозитория
cd my-demo-repo ## заходим
echo "Hello, world" > README.md ## записываем "Hello, world" в README
git add README.md ## добавляем в index
git commit -m "Initial commit" ## создаём commit
git checkout -b feature/big-red-button ## создаём новую ветку и сразу в неё переходим
echo "Big Red Button" > CHANGELOG.md ## записываем "Big Red Button" в CHANGELOG
git add CHANGELOG.md ## добавляем в index
git commit -m "Add CHANGELOG.md" ## создаём commit в ветке feature/big-red-button
git checkout main ## обратно на main, файла CHANGELOG в main нет
git log --oneline --graph --all 
```

## Merge и rebase

Разобравшись с основными понятиями, вернёмся к изучению других терминов.

**Merge (слияние, мёрж)** — это способ объединить два потока изменений. Git создаёт новый коммит, в котором объединены оба направления развития. Это безопасно и понятно, но иногда оставляет лишние следы в истории.

**Rebase** — другой способ объединения. Вместо коммита слияния, Git «переписывает» историю так, будто ваши изменения изначально шли от нужной точки. Чище, но требует аккуратности, дальше в уроке мы подробнее разберёмся, как это работает.

**Pull Request/Merge Request (пулл-реквест, мёрж-реквест)** - запрос на объединение одной ветки с другой. В нём видно, какие изменения попадут в код после слияния, ещё до того, как оно произойдёт. На этом этапе разработчики обычно проводят ревью кода и дают друг другу обратную связь.

И merge, и rebase — не про «лучше/хуже». Merge — вы едете по своей дороге и в конце съезжаете на магистраль: на карте видны обе траектории. Rebase — переноситесь в начало магистрали и проезжаете путь по ней, словно отдельной дороги не существовало. В обоих случаях финиш один, но история путешествия (коммитов) будет разной.

## Зачем нужны стратегии ветвления?

Когда у нас одна ветка — всё понятно. Но как только появляется команда, разные фичи, багфиксы, релизы и эксперименты, возникает вопрос: по каким правилам мы живём?

Вот примеры проблем, которые возникают без стратегии:
- ветка релиза содержит недоделанную фичу;
- один разработчик перезаписывает изменения другого;
- продакшн ломается, потому что кто-то случайно задеплоил незавершённую задачу;
- мёрж-реквесты висят неделями, никто не понимает, что с ними делать.


### Git Flow

В основе Git Flow лежат следующие ветки:
- `main` — содержит только стабильные релизные версии.
- `develop` — основная ветка для интеграции всех фич.
- `feature/*` — ветки для разработки отдельных фич, создаются от `develop`.
- `release/*` — ветки для подготовки релизов, создаются от `develop`, потом вливаются в `main` и `develop`.
- `hotfix/*` — ветки для экстренных исправлений, создаются от `main`.


### GitHub Flow

Как работает GitHub Flow:
- Всё начинается с основной ветки `main`. Она всегда должна быть в рабочем состоянии — её можно деплоить в любой момент.
- Для каждой задачи создаётся новая ветка (например, `feature/login-form` от `main`).
- Работа ведётся в этой ветке. Периодически изменения пушатся, создаётся `pull request`.
- Другие члены команды просматривают изменения, комментируют, предлагают улучшения. После одобрения и прохождения автоматических проверок ветка вливается обратно в `main`. Лучше всего, когда после слияния происходит автоматический деплой.


### GitLab Flow

GitLab Flow стал попыткой объединить формальность Git Flow с гибкостью GitHub Flow и адаптировать это под реальные бизнес-процессы. 

Главная идея GitLab Flow — связать ветки разработки с окружениями и бизнес-контекстом. Это достигается за счёт использования так называемых environment branch или release branch, которые соответствуют средам, а также issue-based ветвления, где каждая задача — это отдельная ветка.

GitLab предлагает три стиля, которые можно комбинировать:
1. Production branch — одна основная ветка (main), релизы деплоятся напрямую оттуда, как в GitHub Flow.
2. Environment branch — ветки вроде staging, preprod, production, куда вливаются фичи перед выкладкой.
3. Release branch — отдельные ветки для каждого релиза, как в Git Flow.

Выбор варианта зависит от зрелости команды, CI/CD-инфраструктуры и процесса QA.

## Какую стратегию выбрать?

Все стратегии рабочие — важно понять, где какая уместнее. Ниже несколько ориентиров для выбора:

1. **Размер и зрелость команды.** Маленькая команда (до 5 человек): GitHub Flow будет оптимальным за счёт простоты и скорости. Средняя команда (5–20 человек): GitLab Flow даёт нужную гибкость без избытка. Большие команды (20+): Git Flow даёт управляемость, особенно если релизы редкие и важна стабильность.
2. **Частота релизов.** Ежедневно или чаще: GitHub Flow или GitLab Flow с автоматическим деплоем. Раз в неделю или реже: GitLab Flow или Git Flow.
3. **Архитектура проекта.** Монолит: любой подход подойдёт, вопрос в дисциплине. Микросервисы: часто используют GitHub Flow или GitLab Flow — каждая команда может адаптировать стратегию под свой сервис.
4. **Наличие среды CI/CD.** Есть полноценный CI/CD: GitHub Flow и GitLab Flow раскрываются полностью. Нет CI/CD: Git Flow проще контролировать вручную, но появляется бюрократия.
5. **Необходимость управления релизами и поддержкой старых версий.** Если нужно поддерживать несколько параллельных релизов (например, разные версии для клиентов), Git Flow всё ещё остаётся одним из немногих удобных решений.
6. **Командная культура и зрелость.** GitHub Flow требует зрелости: тестов, контроля качества, CI. Без этого он может превратиться в хаос. Git Flow навязывает структуру — меньше свободы, но больше предсказуемости. GitLab Flow даёт баланс — его можно настроить под команду и продукт.


## Trunk-based development: когда хочется быстрее и проще

Все рассмотренные выше стратегии (Git Flow, GitHub Flow, GitLab Flow) можно считать представителями одного семейства. У них общая логика: изолируем изменения, работаем в ветках, сливаем через pull/merge request. Это помогает контролировать изменения, но и создаёт определённую инерцию.

**TBD** — это стратегия, при которой все изменения интегрируются как можно быстрее в одну общую ветку (`main` или `trunk`). Здесь нет долгоживущих `feature`-веток. Все работают на одной линии, быстро коммитят, проходят CI, и, если всё в порядке, изменения немедленно вливаются.

Вместо долгоживущих фич, используются техники вроде:
- feature toggle — фича в коде есть, но выключена для пользователей;
- dark deploy — фичу деплоят, но не активируют;
- branch by abstraction — временная прослойка, позволяющая менять реализацию без влияния на остальной код.

**Плюсы Trunk-Based Development:**
- минимум конфликтов: никто не залипает в ветках, всё синхронизировано;
- быстрая обратная связь: CI сразу показывает, что сломано;
- чистота истории: нет лишних merge-коммитов;
- ритм команды: постоянная интеграция формирует дисциплину;

**Минусы и ограничения:**
- невозможно без CI/CD: всё должно быть автоматизировано;
- высокая зрелость команды: нужны культура и опыт;
- фичи не должны ломать продакшн: требуется изоляция через toggle и другие техники скрытия от пользователя фичи;


## Сравнение стратегий: Git Flow, GitHub Flow, GitLab Flow и Trunk-Based Development

Принципы работы:

- **Git Flow:** чёткая структура, долгоживущие ветки, формальный релизный процесс.
- **GitHub Flow:** простота, короткие ветки, быстрое слияние через pull request.
- **GitLab Flow:** гибкость, привязка к окружениям, хорошая интеграция с CI/CD.
- **TBD:** частые коммиты прямо в trunk, минимум ветвления, ставка на автоматизацию и дисциплину.

Подходит, если:
- **Git Flow:** редкие релизы, строгие QA-процессы, необходимость поддержки старых версий.
- **GitHub Flow:** быстрые фичи, зрелая культура ревью, частые выкатки.
- **GitLab Flow:** много окружений, продуктовые команды с разным ритмом, сильная DevOps-инфраструктура.
- **TBD:** максимальная скорость, высокая зрелость команды, CI/CD как норма, минимум ручных операций.

Что же выбрать?
- Для начинающих и небольших команд — GitHub Flow: просто и понятно.
- Для команд с разными продуктами и средами — GitLab Flow: масштабируется и гибко настраивается.
- Для стабильных корпоративных проектов с долгими циклами — Git Flow.
- Для зрелых DevOps-команд с хорошим CI/CD — TBD даёт максимальную отдачу.**


| Характеристика | Git Flow | GitHub Flow | Trunk-Based Development |
|----------------|----------|-------------|-------------------------|
| **Сложность** | Высокая, много веток (`develop`, `release/*`, `hotfix/*`, `feature/*`) | Простая, минимум веток (только `main` и короткие feature-ветки) | Минимальная, почти без веток (работа напрямую в `main` или сверхкороткие ветки < 1 дня) |
| **Релизы** | Через `release/*`, вручную, с фиксированными версиями | Частые, через merge в `main` (обычно несколько раз в день) | Постоянные, автоматические (непрерывная доставка) |
| **CI/CD** | Необязателен, но может использоваться | Требуется (обязательное прохождение тестов перед merge) | Обязателен (быстрый и надёжный CI/CD — критическое условие) |
| **История** | Разветвлённая, много merge-коммитов | Чистая (используется rebase или squash перед merge) | Линейная (минимум merge-коммитов, часто rebase) |
| **Поддержка версий** | Да (`release/*` для релизов, `hotfix/*` для срочных исправлений) | Нет (нет долгоживущих веток для версий) | Нет (все изменения сразу попадают в `main`) |
| **Культура** | Стадии, контроль, выделенная QA | Ревью кода, скорость доставки | Частая интеграция, высокая зрелость команды, парное программирование |
| **Когда применять** | Редкие релизы, формальные процессы, крупные команды | Быстрая разработка с непрерывной доставкой (CD) | Очень зрелые DevOps-команды с мощной автоматизацией |


```bash
git checkout master ## в мастер ветку
git checkout -b feature/small-blue-button ## создали новую и перешли
echo "Add new small blue button" >> CHANGELOG.md ## добавили в CHANGELOG
git add CHANGELOG.md ## в index
git commit -m "Add new small blue button" ## закомитили
git checkout master ## перешли в мастер ветку
echo "Add new line" >> README.md ## добавили в readme
git add README.md ## в index
git commit -m "Add new line" ## закомитили
git checkout feature/small-blue-button ## перешли во feature/small-blue-button
git rebase master ## ребейс - переписали историю, заменили (из ветки фючей)
git checkout master ## перешли в master
git merge feature/small-blue-button ## замёржили (из ветки master)
```

## Ключевые мысли:

1. **Ветки — это фундамент командной разработки.** Они позволяют изолировать изменения, работать параллельно и не мешать друг другу. Без ветвления будет хаос.
2. **Git реализует ветки умно и дёшево.** Ветка — это указатель на коммит. В отличие от старых систем вроде SVN, создание ветки в Git — мгновенная и безопасная операция.
3. **Стратегии ветвления — это соглашения внутри команды.** Git Flow, GitHub Flow, GitLab Flow и Trunk-Based Development — это не только схемы, но и отражение культуры работы.
4. **Каждая стратегия подходит под определённые условия:**
- Git Flow: формально, надёжно, для редких релизов.
- GitHub Flow: быстро, просто, для CI/CD и стартапов.
- GitLab Flow: гибко, масштабируемо, с учётом окружений.
- TBD: радикально быстро, требует зрелости и автоматизации.
5. `merge` **и** `rebase` **— два разных пути объединения.** У каждого — свои случаи применения:
- `merge` — прозрачно, безопасно, но «шумно» в истории.
- `rebase` — чисто, линейно, но требует осторожности.
6. **Стратегия ветвления влияет на процессы.** Она определяет ритм работы, зрелость команды, качество поставки.


## Основные команды Git

### Базовые команды навигации

```shell
pwd                              # Посмотреть, где вы сейчас находитесь
ls -la                           # Посмотреть содержимое текущей папки
cd ~                             # Перейти в домашнюю папку
git status                       # Проверка статуса
```

### Настройка Git

```bash
git config --global user.name "Ваше Имя"
git config --global user.email "ваш@email.com"
```

### Работа с репозиторием

*Репозиторий (repository) - это хранилище вашего проекта вместе с историей изменений.*

```bash
git init                                                 # Создать новый репозиторий
git clone https://github.com/user/repo.git               # Клонировать существующий
```

### Основные команды для коммитов

# Сделать коммит

## Полезные сокращения

Вот список моих любимых команд:
- `git status` - проверяю постоянно 👀
- `git log --oneline` - краткая история коммитов 📜
- `git diff` - смотрю изменения перед коммитом 🔍

### Отмена изменений

```bash
# Отмена изменений в рабочей директории
git checkout -- <файл>
# или
git restore <файл>

# Отмена индексации
git reset HEAD <файл>
# или
git restore --staged <файл>
```

### Получение SHA коммита

*SHA (Secure Hash Algorithm) - это уникальный идентификатор коммита.*

```bash
# Получение полного SHA текущего коммита
git rev-parse HEAD

# Получение короткого SHA
git rev-parse --short HEAD

# Просмотр SHA коммитов в истории
git log --oneline
```

### Сброс (reset)

*Reset позволяет отменить коммиты или изменения в индексе.*

```bash
# Мягкий сброс (сохраняет изменения в рабочей директории)
git reset --soft HEAD~1

# Смешанный сброс (сохраняет изменения, но не в индексе)
git reset HEAD~1

# Жесткий сброс (удаляет все изменения)
git reset --hard HEAD~1
```

### Поиск автора кода (blame)

*Blame позволяет узнать, кто и когда внёс изменения в определённые строки файла.*

```bash
# Просмотр авторов строк в файле
git blame <файл>

# С ограничением по строкам
git blame -L 10,20 <файл>
```

### Поиск в коде (grep)

*Grep позволяет искать строки в файлах репозитория.*

```bash
# Поиск строки во всех файлах
git grep "строка_для_поиска"

# Поиск с учетом контекста
git grep -n "строка_для_поиска"

# Поиск только в определенных файлах
git grep "строка_для_поиска" -- "*.js"
```

## Работа с ветками

### Слияние (merge)

*Merge объединяет изменения из одной ветки в другую.*

```bash
# Слияние ветки в текущую
git merge <имя_ветки>

# Слияние с созданием коммита слияния (no fast-forward)
git merge --no-ff <имя_ветки>

# Отмена слияния
git merge --abort
```

#### Fast-forward (FF) vs No Fast-forward (no-FF)

- **Fast-forward (FF)**: Если возможно, Git просто перемещает указатель текущей ветки вперед. Не создаётся отдельный коммит слияния.

- **No Fast-forward (no-FF)**: Всегда создаётся коммит слияния, даже если можно выполнить fast-forward. Это сохраняет информацию о существовании ветки в истории.

### Перебазирование (rebase)

*Rebase переносит коммиты из одной ветки на конец другой, создавая линейную историю.*

```bash
# Перебазирование текущей ветки на другую
git rebase <имя_ветки>

# Отмена перебазирования
git rebase --abort

# Продолжение перебазирования после разрешения конфликтов
git rebase --continue
```

#### Интерактивный rebase

*Интерактивный rebase позволяет изменять, объединять, удалять и переупорядочивать коммиты.*

```bash
# Интерактивный rebase последних N коммитов
git rebase -i HEAD~N

# Интерактивный rebase от определенного коммита
git rebase -i <хеш_коммита>^
```

В открывшемся редакторе вы можете:
- `pick` - оставить коммит как есть
- `reword` - изменить сообщение коммита
- `edit` - остановиться для изменения коммита
- `squash` - объединить с предыдущим коммитом
- `fixup` - объединить с предыдущим коммитом, отбросив сообщение
- `drop` - удалить коммит

### Объединение коммитов (squash)

*Squash позволяет объединить несколько коммитов в один.*

```bash
# Объединение последних N коммитов через интерактивный rebase
git rebase -i HEAD~N
# Затем замените "pick" на "squash" или "fixup" для коммитов, которые нужно объединить
```

## Git Hooks (хуки)

*Хуки - это скрипты, которые Git запускает перед или после определенных событий.*

Основные хуки:
- `pre-commit`: запускается перед созданием коммита
- `prepare-commit-msg`: запускается перед открытием редактора сообщения коммита
- `commit-msg`: запускается для проверки сообщения коммита
- `post-commit`: запускается после создания коммита
- `pre-push`: запускается перед отправкой изменений
- `post-checkout`: запускается после переключения веток

```bash
# Хуки хранятся в директории .git/hooks/
# Пример создания pre-commit хука
echo '#!/bin/sh
# Проверка стиля кода
npm run lint' > .git/hooks/pre-commit
chmod +x .git/hooks/pre-commit
```

## Дополнительные команды

### Stash

*Stash позволяет временно сохранить изменения, не создавая коммит.*

```bash
# Сохранение изменений
git stash

# Применение последнего stash
git stash apply

# Удаление последнего stash после применения
git stash pop

# Просмотр списка stash
git stash list

# Удаление stash
git stash drop stash@{n}
```

### Cherry-pick

*Cherry-pick позволяет применить изменения из одного коммита к текущей ветке.*

```bash
# Применение коммита к текущей ветке
git cherry-pick <хеш_коммита>

# Применение нескольких коммитов
git cherry-pick <хеш1> <хеш2>
```

### Reflog

*Reflog хранит историю изменений указателей (HEAD, ветки) и позволяет восстановить потерянные коммиты.*

```bash
# Просмотр истории изменений HEAD
git reflog

# Восстановление до определенного состояния
git reset --hard HEAD@{n}
```

### Bisect

*Bisect помогает найти коммит, в котором появилась ошибка, с помощью бинарного поиска.*

```bash
# Начало бинарного поиска
git bisect start

# Отметка текущего коммита как плохого
git bisect bad

# Отметка определенного коммита как хорошего
git bisect good <хеш_коммита>

# Отметка текущего коммита как хорошего/плохого в процессе поиска
git bisect good
git bisect bad

# Завершение поиска
git bisect reset
```
