# Разбор pipeline
## Рассмотрим учебный репозиторий

Учебный проект для урока 2 модуля 2. Веб-приложение с Flask-бэкендом и Vue.js-фронтендом — используется как сквозной пример при изучении архитектуры CI-пайплайна.

```
my-app/
├── backend/              # Flask API (Python)
│   ├── app.py
│   ├── requirements.txt
│   └── tests/
│       └── test_app.py
├── frontend/             # Vue.js-приложение (Vite)
│   ├── src/
│   │   ├── App.vue
│   │   ├── App.test.js
│   │   └── main.js
│   ├── index.html
│   ├── package.json
│   └── vite.config.js
└── .gitlab-ci.yml
```

```yaml
# Указываем Docker-образ, в котором будут выполняться все задания.
# Python 3.11 — это среда с установленным интерпретатором и pip.
image: python:3.11

# Определяем этапы выполнения пайплайна в хронологическом порядке.
# Задания внутри одного этапа выполняются параллельно.
stages:
  - lint      # Проверка качества кода (линтеры)
  - build     # Сборка/установка зависимостей
  - test      # Запуск тестов

# Настройка кэширования — сохраняет файлы между запусками пайплайна.
# Это ускоряет выполнение, т.к. не нужно переустанавливать зависимости.
cache:
  key: ${CI_COMMIT_REF_SLUG}  # Уникальный ключ кэша для каждой ветки
  paths:                       # Какие папки сохранять
    - backend/.venv/           # Виртуальное окружение Python
    - backend/__pycache__/     # Скомпилированные файлы Python
    - .cache/pip               # Кэш загрузок pip

# Команды, которые выполняются перед каждым заданием.
# Используется для общих подготовительных действий.
before_script:
  - cd backend                 # Переходим в папку с backend-кодом
  - pip install --cache-dir .cache/pip  # Устанавливаем pip с кэшированием

# ========== ЗАДАНИЕ 1: Flake8 ==========
# Проверяет стиль кода (PEP8), неиспользуемые импорты и т.д.
flake8_lint:
  stage: lint                   # Относится к этапу lint
  script:                       # Команды, которые выполняются в задании
    - pip install flake8        # Устанавливаем линтер
    - flake8 src/               # Запускаем проверку папки src/

# ========== ЗАДАНИЕ 2: Pylint ==========
# Более глубокая проверка кода: имена переменных, сложность функций и т.д.
pylint_check:
  stage: lint
  script:
    - pip install pylint flask  # Устанавливаем pylint с зависимостями
    - pylint src/ --fail-under=8.0  # Ошибка, если оценка < 8/10

# ========== ЗАДАНИЕ 3: Bandit ==========
# Поиск уязвимостей безопасности в коде.
bandit_security:
  stage: lint
  script:
    - pip install bandit        # Устанавливаем bandit
    - bandit -r src/ -ll        # -r: рекурсивно, -ll: средние+высокие угрозы

# ========== ЗАДАНИЕ 4: Сборка ==========
# Устанавливает зависимости проекта.
build_app:
  stage: build
  script:
    - pip install -r requirements.txt  # Устанавливаем зависимости из файла
  artifacts:                    # Что сохранить после выполнения задания
    paths:                      # Какие файлы/папки сохранить
      - backend/.venv/          # Виртуальное окружение
    expire_in: 1 hour           # Через час артефакты удалятся

# ========== ЗАДАНИЕ 5: Тесты ==========
# Запускает pytest с проверкой покрытия кода.
run_tests:
  stage: test
  script:
    - pip install pytest pytest-cov  # Устанавливаем инструменты тестирования
    - pytest tests/ --maxfail=1 -q --cov=src --cov-report=term
    # --maxfail=1: остановиться после первой ошибки
    # -q: краткий вывод
    # --cov=src: проверять покрытие кода в папке src
    # --cov-report=term: вывести отчёт в терминал
  dependencies:                # Указываем, от каких заданий зависят артефакты
    - build_app                # Берём .venv из задания build_app
```

## Краткое описание пайплайна
Этот пайплайн автоматизирует проверку качества Python-кода. Он выполняет три этапа:

1. **Линтинг (lint)** — три параллельных задания проверяют код разными инструментами:
    - `flake8` — стиль и форматирование
    - `pylint` — логические ошибки и качество
    - `bandit` — уязвимости безопасности
2. **Сборка (build)** — устанавливает зависимости проекта из requirements.txt
3. **Тестирование (test)** — запускает pytest с проверкой покрытия кода

Если какое-либо задание падает, пайплайн останавливается (кроме параллельных заданий на этапе lint — они выполняются независимо).

## Основная терминология

**Pipeline** - Последовательность автоматических шагов при каждом пуше в репозиторий

**Stage** - Группа заданий, выполняемых параллельно. Стадии идут по порядку

**Job**	- Одна единица работы (например, запуск flake8)

**Image** - Docker-контейнер с предустановленным окружением (Python, Node.js и т.д.)

**Script** - Команды, которые выполняются внутри задания

**Cashe** - Файлы, сохраняемые между запусками для ускорения

**Artifacts** - Файлы, которые сохраняются после выполнения задания (отчёты, логи, сборки)

**Path** - Относительный путь к файлам/папкам для кэша или артефактов

**Cache key** - Идентификатор, чтобы разные ветки имели разные кэши

**Dependencies** - Зависимости/указание, из какого задания брать артефакты

**Variable** - CI/CD переменные (например, CI_COMMIT_REF_SLUG — имя ветки)

## Важные замечания из реальной практики

| Ситуация | Решение |
| :--- | :--- |
| Задание упало, но артефакты нужны | Добавить `when: always` в секцию `artifacts` |
| Задание использует другой образ | Указать `image: node:18` (или другой) внутри задания |
| Нужно пропустить задание для некоторых веток | Использовать `only:` или `except:` |
| Переменная доступна во всех заданиях | Определить `variables:` на верхнем уровне |
| Команда повторяется во многих заданиях | Вынести в `before_script` |
| Задание зависит от артефактов другого задания | Использовать `dependencies:` с указанием имени задания |
| Нужно сохранить отчёты, даже если тесты упали | Добавить `when: always` в `artifacts` |
| Запускать задание только на определённой ветке | Добавить `only: - main` |
| Избежать двойной установки зависимостей | Использовать `cache` + `dependencies` |
| Задать таймаут для долгого задания | Добавить `timeout: 10 minutes` |
