## Структура CI-пайплайна в GitLab CI

В GitLab CI весь процесс описывается в специальном файле `.gitlab-ci.yml`, который лежит в корне репозитория. Именно там мы указываем, что нужно делать, в каком порядке и при каких условиях. Условия старта пайплайна могут быть разными: по умолчанию это новый коммит в ветку, но можно настроить запуск по расписанию или по API-запросу.

Чтобы описать процесс понятным и управляемым способом, GitLab делит его на две основные части: stages и jobs.

**Stage (этап)** — это крупный логический шаг. В CI — сначала проверка линтером, потом сборка, затем тесты и в конце — деплой.

GitLab выполняет stages строго по порядку, один за другим. Следующий stage не начнётся, пока все задания (jobs) текущего не завершатся успешно. Это помогает сохранить структуру: если линтер нашёл ошибки, нет смысла тратить ресурсы на сборку.

Внутри каждого stage выполняются одно или несколько заданий (jobs). job — это конкретная инструкция: что делать, какую команду запустить и с какими параметрами. Jobs одного stage можно выполнять параллельно, если они не зависят друг от друга: сборка бекенда и фронтенда может работать одновременно — каждый собирает свою часть приложения независимо.

Итак, в GitLab CI stages задают последовательность, а jobs внутри stages — конкретные шаги. Такой подход помогает разделить процесс на управляемые части: видно, где что выполняется и на каком этапе возникла ошибка.

Давайте визуализируем структуру:

```yaml
stages:
  - build
  - test

build_backend:
  stage: build
  script:
    - // какая-то команда для сборки бекенда
  
build_front:
  stage: build
  script:
    - // какая-то команда для сборки фронта
  
build_test:
  stage: test
  script:
    - // какая-то команда для тестирования бекенда
  
build_test:
  stage: test
  script:
    - // какая-то команда для тестирования фронта 
```

Тут у нас два stage: build, test и четыре jobs: build_backend, build_front, build_test, build_test.

### Где и как GitLab запускает jobs

Когда мы описываем пайплайн (от англ. pipeline — «трубопровод») в `.gitlab-ci.yml`, кажется, что всё «просто работает»: команды выполняются, артефакты передаются, jobs идут по порядку. Но где именно всё это происходит? GitLab сам по себе не запускает jobs — для этого у него есть отдельный компонент **GitLab Runner**.

GitLab Runner — это агент, который подключается к GitLab и выполняет jobs из `.gitlab-ci.yml`. Когда мы пушим код, GitLab передаёт инструкции раннеру, и тот запускает всё, что нужно: команды, сборки, тесты, деплой и всё, что описано в нашем пайплайне.
Раннер может работать на разных типах инфраструктуры:
- на обычной виртуальной машине (например, в облаке);
- в Docker-контейнере;
- внутри Kubernetes-кластера;
- на реальном сервере внутри компании;
- на Linux, Windows и даже macOS.

Что именно делает раннер? Он берёт наш job, выполняет описанные в нём команды, собирает результаты и отправляет обратно в GitLab статус (успех или ошибка) плюс логи выполнения.

*Каждый job пайплайна исполняется изолированно, в своей среде, однако между вашими job'ами вы можете передавать артефакты и переменные, даже если они исполняются на разных раннерах*

К Gitlab можно подключить сколько угодно раннеров, на них можно вешать метки (например, по операционной системе), их можно делать выделенными для проекта или общими для всего Gitlab.
Для начала важно просто знать: `.gitlab-ci.yml` — это инструкция, которую выполняет агент в вашей или общей инфраструктуре.

## Stages и зависимости

### Логическое разбиение на stages

Мы уже говорили, что этапы (stages) задают общий порядок выполнения jobs в CI. Но важно понять, зачем вообще делить работу на этапы, а не просто писать список команд сверху вниз.
Ответ можно найти в философии DevOps. В ней жизненный цикл продукта — это не линейная прямая от идеи до релиза, а непрерывный цикл от планирования до мониторинга и обратно. Такой цикл часто рисуют в виде восьмёрки:

`Plan → Code → Build → Test → Release → Deploy → Operate → Monitor → (Plan)`

CI-пайплайн по сути реализует левую половину этой восьмёрки: от написания кода до релиза. Stages вроде `build`, `test`, `release` — это не случайный набор слов, а техническая реализация ключевых фаз разработки.
При этом результаты мониторинга и эксплуатации (Operate → Monitor) возвращаются в планирование и новые коммиты, замыкая цикл. Правильное проектирование пайплайна со stages, следующими этим шагам, не только ускоряет процесс, но и поддерживает культуру непрерывных улучшений и обратной связи DevOps.

*Не стоит дробить пайплайн на слишком много последовательных stages без необходимости. Каждый stage ждёт окончания предыдущего, и чем их больше, тем потенциально длиннее пайплайн. Логика разбиения должна отражать смысловые и технические зависимости.*

### Stages по умолчанию

В Gitlab CI по умолчанию есть набор stages: `.pre`, `build`, `test`, `deploy`, `.post`. Мы можем использовать его, а можем задать свои имена и свой порядок. Как мы говорили, все задания (jobs) внутри одного stage выполняются параллельно, а stage считается завершённым, только когда все запущенные jobs закончились. После этого стартует следующий stage.

Например:
```yaml
stages:
  - lint
  - build
  - test
  - deploy 
 ```

 Порядок `lint → build → test` типичен для компилируемых языков (Java, Go, C#), где тесты требуют собранных артефактов. В интерпретируемых языках (Python, Node.js без отдельной сборки) юнит-тесты иногда запускают до сборки — `lint → test → build`.

 Например:
```yaml
lint_backend:
  stage: lint
  script:
    - cd backend
    - pip install flake8
    - flake8 src/

lint_frontend:
  stage: lint
  script:
    - cd frontend
    - npm ci
    - npm run lint

build_backend:
  stage: build
  script:
    - cd backend
    - pip install -r requirements.txt

build_frontend:
  stage: build
  script:
    - cd frontend
    - npm ci
    - npm run build 
```

Здесь оба jobs линтинга запускаются параллельно первыми. Только после их завершения запустятся два jobs по сборке: `build_backend` и `build_frontend`. Они принадлежат одному этапу и тоже могут выполняться параллельно. Это упрощает проектирование, но иногда приводит к лишнему ожиданию. Например, если есть две независимые цепочки (бэкенд и фронтенд), они всё равно вынуждены дожидаться окончания всего предыдущего этапа.
Представьте: `build_backend` занимает 5 минут, а `build_frontend` — 30 секунд. Тесты фронтенда будут ждать 5 минут, хотя их зависимость готова уже через 30 секунд. Для таких случаев есть специальный инструмент `needs`.

### Needs — явные зависимости

Чтобы сделать пайплайн ещё быстрее, в GitLab CI есть директива `needs`. Она позволяет явно задать зависимости между jobs: тогда GitLab CI запустит job сразу после завершения тех jobs, от которых он зависит, не дожидаясь конца всего stage.
Пример:

```yaml
stages:
  - build
  - test
  - deploy

build_a:
  stage: build
  script: echo "Building A"

build_b:
  stage: build
  script: echo "Building B"

test_a:
  stage: test
  script: echo "Testing A"
  needs: [build_a]

test_b:
  stage: test
  script: echo "Testing B"
  needs: [build_b]

deploy_a:
  stage: deploy
  script: echo "Deploying A"
  needs: [test_a]

deploy_b:
  stage: deploy
  script: echo "Deploying B"
  needs: [test_b] 
```

В этом примере:
- `test_a` начнётся сразу после `build_a`, не дожидаясь `build_b`;
- аналогично, `deploy_a` стартует сразу после `test_a`, не ожидая завершения `test_b`.
Это сильно сокращает общее время пайплайна, если, например, `build_b` и `test_b` очень долгие, а `A` — быстрые.
Механизм `needs` ускоряет процесс, но требует аккуратности. Если все jobs переплести зависимостями, пайплайн станет запутанным. Трудно будет понять, где что сломалось, а визуализация в GitLab превратится в паутину.

*Особый случай — `needs: []` (пустой список). Это явное указание: «запусти этот job немедленно, не дожидаясь завершения ни одного предыдущего stage». Job стартует в самом начале пайплайна параллельно с первым stage. Это удобно для полностью независимых задач, но нарушает принцип Fail Fast: если такой job стартует до линтера, пайплайн продолжит работу даже если линтер упадёт.*

Иногда в следующем stage пайплайна нужно использовать результаты предыдущего. Например, это может быть собранный архив, бинарный файл или отчёт о тестах. В GitLab CI для этого есть механизм **артефактов (artifacts)** — они позволяют передавать файлы от одного job к другому. Если используется `needs`, GitLab сам позаботится о том, чтобы нужные данные были доступны:

```yaml
build_app:
  stage: build
  script: make build
  artifacts:
    paths: ["dist/"]

test_app:
  stage: test
  script: ./run-tests
  needs:
    - build_app 
```

Здесь test_app может сразу использовать результаты сборки (dist/), не дожидаясь других сборок, не ища файлы вручную и не скачивая что-то снаружи — GitLab автоматически скачивает артефакты из всех jobs, указанных в needs.


# Принцип Fail Fast в CI-пайплайне

**Fail Fast** — это подход, при котором ошибки выявляются как можно раньше, чтобы не тратить время и ресурсы на ненужные дальнейшие шаги. Лучше узнать о проблеме через 30 секунд, чем через 30 минут после долгих сборок и тестов.

Этот подход:
- экономит ресурсы CI/CD — меньше вычислительных ресурсов и меньше лишних артефактов;
- даёт обратную связь разработчикам быстрее;
- снижает время простоя команды: разработчики быстрее находят и исправляют баги — быстрее возвращаются к нормальной работе.
Чтобы пайплайн падал быстро, оперативные и чувствительные проверки нужно выносить в самое начало. GitLab CI выполнит stages в указанном порядке, и если на раннем stage что-то пошло не так — всё остановится, остальные jobs запускаться не будут.
Например, в пайплайне:
```yaml
stages:
  - lint
  - build
  - test
  - deploy

lint_code:
  stage: lint
  script:
    - npm run lint
```

Если линтер нашёл ошибки, на сборку и тесты время тратиться не будет. Это особенно важно в больших проектах, где тесты или билд могут занимать по 5–10 минут. Лучше остановиться в самом начале, до того, как будут потрачены ресурсы всей команды.

## Что ещё стоит учитывать
### Группируйте задачи осмысленно

Например, линтинг и статический анализ можно вынести в отдельный stage, который всегда идёт первым. Это помогает визуально и логически отделить «быстрые проверки» от «тяжёлых задач».

### Needs может мешать Fail Fast

Если используется `needs` для запуска цепочек параллельно, нужно быть осторожными — GitLab по умолчанию **не отменяет другие jobs**, если один упал. Например, в пайплайне ниже, если `build_a` упадёт, `test_a` просто не запустится, но `build_b` и `test_b` продолжат выполняться. Это может быть желаемым, а может — нет.

```yaml
stages:
  - build
  - test

build_a:
  stage: build
  script: make A

build_b:
  stage: build
  script: make B

test_a:
  stage: test
  script: ./test A
  needs: [build_a]

test_b:
  stage: test
  script: ./test B
  needs: [build_b] 
```

### Не злоупотребляйте allow_failure

GitLab позволяет пометить job как необязательный (`allow_failure: true`). Это может быть полезно для экспериментальных шагов, но если сделать необязательными ключевые проверки, Fail Fast просто не сработает.

```yaml
test_noncritical:
  script: ./test-noncritical
  allow_failure: true 
```

Это значит, что если этот job упадёт, пайплайн всё равно продолжит выполняться и будет зелёным.

## Примеры пайплайнов с Fail Fast
### Хороший пример
```yaml
stages:
  - lint
  - build
  - test

lint_code:
  stage: lint
  script:
    - npm run lint

build_app:
  stage: build
  script:
    - npm run build

unit_tests:
  stage: test
  script:
    - npm run test 
```
Пайплайн упадёт сразу, если линтинг не пройден. Build и тесты не начнутся впустую.

### Плохой пример

```yaml
stages:
  - build
  - test
  - lint

build_app:
  stage: build
  script:
    - npm run build

run_tests:
  stage: test
  script:
    - npm run test

lint_code:
  stage: lint
  script:
    - npm run lint 
```
Линтер запускается в самом конце, после сборки и тестов. Значит, пайплайн может прогнать долгие тесты, а затем упасть на форматировании кода. Время будет потрачено зря.
Итак, Fail Fast — это не про строгость, а про уважение ко времени: и вашему, и всей команды. Он помогает сократить цикл обратной связи, быстрее находить ошибки и быстрее возвращаться к работе.


# Переиспользование кода и оптимизация CI-пайплайна
## Якоря YAML (& и *)

Это стандартная возможность YAML, не специфичная для GitLab CI. С помощью якоря можно один раз описать повторяющийся блок, а потом вставлять его в другие места. Например, так:
```yaml
.default_script: &default_script
  script:
    - npm ci
    - npm run lint

lint_code:
  stage: lint
  <<: *default_script 
```

## Extends

GitLab CI предлагает более наглядный способ переиспользования — механизм `extends`. Он позволяет описать шаблон job и наследовать эти же настройки в других jobs.
```yaml
.default-job:
  script:
    - npm ci
  cache:
    paths: [node_modules/]

build:
  stage: build
  extends: .default-job
  script:
    - npm run build

lint:
  stage: lint
  extends: .default-job
  script:
    - npm run lint 
```
В этом примере и `build`, и `lint` наследуют настройки из `.default_job`, но могут переопределять конкретные команды. Это удобнее и читаемее, чем YAML-якоря.

## Include

Когда шаблонов становится много, их можно вынести в отдельный файл. Например, если в организации используется единый шаблон линтинга, его удобно хранить в отдельном репозитории и подключать через `include`. Например:
```yaml
include:
  - project: 'my-group/shared-ci'
    file: '/templates/nodejs.yml' 
```
Можно подключать как собственные шаблоны, так и готовые из GitLab (например, для SAST или DAST):
```yaml
include:
  - template: Security/SAST.gitlab-ci.yml 
```
Это помогает унифицировать процессы между командами и избавляет от дублирования.

## Переиспользование результатов работы

Как мы упоминали ранее, если один job создаёт файлы, которые нужны следующему — например, сборка проекта, которая потом пойдёт в тесты или деплой, — их можно передать с помощью `artifacts`.
```yaml
build:
  stage: build
  script: make build
  artifacts:
    paths: [dist/]

test:
  stage: test
  script: ./run-tests
  needs: [build] 
```

Если вы используете `needs`, артефакты передаются автоматически. В противном случае можно явно указать `dependencies`.

Еще один очень удобный инструмент — `cache`. Это способ сохранить данные между разными пайплайнами. Особенно полезен для зависимостей: `node_modules/`, `.m2/`, `venv/` и других.
```yaml
cache:
  key: ${CI_COMMIT_REF_SLUG}
  paths:
    - node_modules/ 
```

Такой подход экономит время на повторную установку зависимостей, особенно если они меняются нечасто.
Как GitLab работает с кешем: после завершения job Runner сохраняет указанные пути по заданному ключу. При следующем запуске — восстанавливает их до выполнения `script`. Если кеша нет (первый запуск или изменился ключ), job работает без него и создаёт новый.
Отличие от `artifacts`: `cache` предназначен для переиспользования между пайплайнами в рамках одной ветки — это оптимизация скорости. artifacts — для передачи результатов между jobs внутри одного пайплайна (например, скомпилированный `.jar` из `build` в `test`). Не путайте: кеш — для ускорения, артефакты — для данных.
Жизненный цикл кеша: кеш инвалидируется при смене ключа. Сменился ключ (например, обновился `package-lock.json` в составе ключа) — кеш пересоздаётся с нуля. GitLab автоматически удаляет устаревшие кеши по настраиваемому периоду хранения.
Два полезных варианта наименования ключа:

```yaml
# Кеш по ветке — каждая ветка получает свой независимый кеш
cache:
  key: ${CI_COMMIT_REF_SLUG}
  paths:
    - node_modules/

# Общий кеш для всего проекта — все ветки используют один кеш (удобно для стабильных зависимостей)
cache:
  key: ${CI_PROJECT_NAME}
  paths:
    - node_modules/ 
```

Выбор ключа зависит от задачи: ключ по ветке (`CI_COMMIT_REF_SLUG`) изолирует кеши между ветками, ключ по проекту (`CI_PROJECT_NAME`) экономит место, но может приводить к гонке кешей при параллельных пайплайнах.
Итак, продуманная архитектура пайплайна и переиспользование — это про заботу о будущем. Когда пайплайн прост, можно обойтись и без шаблонов. Но когда задач становится десятки, правильная организация приобретает такую же важность, как структура кода в проекте. Если шаблон установки зависимостей или деплоя используется в нескольких jobs — опишите его один раз, а не копируйте. Так пайплайн остаётся компактным и понятным.


# Ключевые мысли
- Пайплайн GitLab CI состоит из stages (этапов) и jobs (заданий). Stages определяют последовательность крупных фаз процесса (build → test → deploy), а jobs внутри stages выполняются параллельно.
- Fail Fast — выявляйте ошибки как можно раньше. Начинайте пайплайн с быстрых проверок (линтинг, статический анализ), чтобы быстро остановить процесс при проблемах и не тратить ресурсы на бессмысленные сборки и тесты.
- Используйте `needs` для управления зависимостями и ускорения. `needs` позволяет запускать jobs сразу после готовности зависимостей, минуя строгую последовательность stages. Это сокращает общее время выполнения пайплайна.
- Передавайте результаты через `artifacts`. С помощью `artifacts` (и `dependencies`, либо автоматически при `needs`) можно передавать результаты работы между jobs.
- Используйте `cache` для ускорения установки зависимостей между пайплайнами. Например, кешировать `node_modules` или `Maven .m2`.
- Переиспользуйте код в CI:
    - используйте YAML-якоря (`&` и `*`) для повторяющихся блоков;
    - применяйте `extends` для наследования шаблонов в GitLab CI;
    - подключайте `include`, чтобы использовать общие шаблоны между проектами.
- Проектируйте пайплайн как инженерный продукт. Хорошо продуманная архитектура CI/CD делает процесс стабильным, понятным и легко масштабируемым.


# Автоматическая проверка качества
## Как мы можем проверять качество кода?

Всё, что связано с качеством кода, можно условно разделить на два больших класса проверок:
1. **Работоспособность кода** — здесь нас интересует, что код действительно выполняет задуманную логику и новый код не ломает старый. Для этого применяют разные виды автоматических тестов, которые запускают программу или её части и проверяют результаты.
2. **Корректность кода и его безопасность** — тут нам важно проверить, что код читаемый, следует общему стилю проекта, не содержит типичных ошибок и потенциальных уязвимостей, а функции и методы используются с правильными типами данных. Для таких целей применяют инструменты статической проверки, которые анализируют код как текст без запуска. Иногда можно встретить название «статические анализаторы».

### Тесты

По типам тесты можно разделить на:
- **юнит-тесты** — проверяют отдельные маленькие кусочки кода: функции или классы;
- **интеграционные тесты** — убеждаются, что модули корректно взаимодействуют между собой;
- **E2E (end-to-end) тесты** — проходят весь путь, как пользователь: от запроса на фронтенде до обновления в базе. Именно этими тестами часто проверяют пользовательский сценарий целиком.

### Статический анализ

В отличие от тестов, статический анализ **не запускает код**, а читает его как текст и строит «картину» приложения.

Как правило, выделяют несколько типов:
- **линтеры** — проверяют стиль, структуру, потенциальные ошибки: «ругаются» на лишние импорты, странные имена переменных, пропущенные docstrings;
- **анализаторы типов** — проверяют, что функции и переменные используются так, как описано в type hints.
- **инструменты SAST** — ищут уязвимости в безопасности, например прямую конкатенацию строк в SQL-запросах.

*Статический анализ может найти ошибки в коде, который не покрыт тестами.*

### Почему нужны оба подхода вместе

Автоматические тесты и статический анализ отлично дополняют друг друга:
- **тесты** дают уверенность, что новый код делает то, что от него ждут, и не ломает старые сценарии;
- **статический анализ** гарантирует, что код написан качественно и не содержит скрытых багов — даже в местах, которые пока не покрыты тестами.

В реальных CI-пайплайнах всегда ставят **оба вида проверок**. Обычно сначала запускается статический анализ (он быстрый и сразу ловит грубые ошибки), а затем идут тесты. 
Такой порядок реализует принцип **Fail Fast**: пусть пайплайн падает как можно раньше, если в коде есть очевидные ошибки.

## Инструменты статической проверки на примере Python

Разберём популярные инструменты для Python, которые чаще всего встраивают в CI для реализации тестирования и статического анализа.

### Линтеры и анализаторы типов

Линтеры читают код и сразу подсвечивают потенциальные ошибки, подозрительные места или нарушения стиля. Они могут находить:
- неиспользуемые переменные и импорты;
- нарушения PEP 8 (отступы, длина строк и др.);
- использование переменной до её объявления;
- вызов функции с неправильным числом аргументов;
- сложные ветвления, которые могут скрыть баги.

Ещё часто используют термин **code smell** — это набор определённых анализатором кода правил, который выводит конечную оценку для вашего кода.
Самый известный линтер для Python — **Flake8**: классика для проверки PEP 8 и мелких огрехов. С помощью плагинов он расширяется до поиска потенциальных ошибок. В качестве альтернативы существует более современный инструмент Ruff.

Для более глубокого анализа принято использовать **Pylint** — он проверяет стиль и пытается «понять» код. Pylint может указать на вызов функции с неверным числом аргументов или использование неопределённой переменной.

В CI-пайплайне обычно сначала запускают быстрый линтер (Flake8), затем более глубокий анализ (Pylint). Если что-то из них упадёт, пайплайн сразу остановится — не придётся тратить время на тесты, если код уже явно не очень чистый.

### SAST

Для статического анализа безопасности в Python чаще всего используют **Bandit**. Это сканер, который ищет небезопасные конструкции в исходниках:
- прямое чтение конфигов с паролями;
- использование `assert` в проде;
- небезопасное использование `pickle`;
- возможные SQL-инъекции через конкатенацию строк.

**Место в пайплайне:** stage `lint` — Bandit не запускает код, а читает его как текст. Это значит, что его можно запустить до сборки и тестов, рядом с Flake8 и Pylint.

Итого, что делает каждый инструмент и в каком stage:

**Flake8** - Нарушения PEP 8, длинные строки, неиспользуемые импорты - `lint`
**Pylint** - Вызовы с неверным числом аргументов, неинициализированные переменные, code smell - `lint`
**Bandit**- SQL-инъекции, небезопасный pickle, захардкоженные пароли - `lint`
**Pytest**- Функциональность кода: юнит и интеграционные тесты - `test`
**coverage.py** - Процент покрытия тестами — показывает, какой код вообще не запускается в тестах - `test`

## Автоматические тесты

### Юнит-тесты

Юнит-тесты — это маленькие тесты, которые проверяют работу отдельных функций или классов. Они максимально изолированы: никакой базы данных, никаких файлов на диске (если можно), только чистая логика.

Для проверки работоспособности Python-кода чаще всего используют **Pytest**. Этот инструмент автоматически распознаёт, где тесты, а где обычные функции, благодаря простому соглашению:
- Pytest ищет все файлы, которые начинаются с `test_` или заканчиваются на `_test.py`;
- внутри таких файлов он ищет функции, которые называются `test_*`.

### Интеграционные тесты

Интеграционные тесты проверяют, как разные модули системы работают вместе. Например, что функция, которая сохраняет данные в базу, потом действительно может их вернуть. Для таких тестов часто используют временную тестовую БД.
Для Python такие тесты обычно тоже пишут через Pytest, но с использованием тестовой базы данных или моков. Иногда в CI для интеграционных тестов запускают контейнеры с PostgreSQL или Redis, чтобы проверить боевое взаимодействие.

### E2E (end-to-end)

E2E-тесты — это полноценные сценарии с точки зрения пользователя. Например:
- открыть браузер, зайти на страницу входа, ввести логин/пароль и проверить, что попали в личный кабинет;
- отправить HTTP-запрос через `requests` к запущенному API и проверить, что ответ верный.

Для этого обычно используют инструменты:
- Selenium — автоматизация браузера для web UI;
- Cypress — тоже для web, с удобной отладкой;
- Playwright — современная альтернатива Selenium с поддержкой нескольких браузеров;
- Pytest + requests — для API, если тестируем REST напрямую.

End-to-end тесты крайне полезны, но они тяжелее и медленнее других типов, потому что:
- запускают настоящие браузеры или полноценное приложение;
- требуют поднятых всех сервисов системы;
- тестируют длинные сценарии, что занимает минуты (иногда десятки минут).
По этой причине такой тип тестов ставят в конец CI-пайплайна, а в ряде случаев запускают отдельно — по ночам или выходным, например. Зато именно они дают наибольшую уверенность, что приложение корректно работает с точки зрения пользователя.
