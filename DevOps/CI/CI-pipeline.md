## Структура CI-пайплайна в GitLab CI

В GitLab CI весь процесс описывается в специальном файле `.gitlab-ci.yml`, который лежит в корне репозитория. Именно там мы указываем, что нужно делать, в каком порядке и при каких условиях. Условия старта пайплайна могут быть разными: по умолчанию это новый коммит в ветку, но можно настроить запуск по расписанию или по API-запросу.

Чтобы описать процесс понятным и управляемым способом, GitLab делит его на две основные части: stages и jobs.

**Stage (этап)** — это крупный логический шаг. В CI — сначала проверка линтером, потом сборка, затем тесты и в конце — деплой.

GitLab выполняет stages строго по порядку, один за другим. Следующий stage не начнётся, пока все задания (jobs) текущего не завершатся успешно. Это помогает сохранить структуру: если линтер нашёл ошибки, нет смысла тратить ресурсы на сборку.

Внутри каждого stage выполняются одно или несколько заданий (jobs). job — это конкретная инструкция: что делать, какую команду запустить и с какими параметрами. Jobs одного stage можно выполнять параллельно, если они не зависят друг от друга: сборка бекенда и фронтенда может работать одновременно — каждый собирает свою часть приложения независимо.

Итак, в GitLab CI stages задают последовательность, а jobs внутри stages — конкретные шаги. Такой подход помогает разделить процесс на управляемые части: видно, где что выполняется и на каком этапе возникла ошибка.

Давайте визуализируем структуру:

```yaml
stages:
  - build
  - test

build_backend:
  stage: build
  script:
    - // какая-то команда для сборки бекенда
  
build_front:
  stage: build
  script:
    - // какая-то команда для сборки фронта
  
build_test:
  stage: test
  script:
    - // какая-то команда для тестирования бекенда
  
build_test:
  stage: test
  script:
    - // какая-то команда для тестирования фронта 
```

Тут у нас два stage: build, test и четыре jobs: build_backend, build_front, build_test, build_test.

### Где и как GitLab запускает jobs

Когда мы описываем пайплайн (от англ. pipeline — «трубопровод») в `.gitlab-ci.yml`, кажется, что всё «просто работает»: команды выполняются, артефакты передаются, jobs идут по порядку. Но где именно всё это происходит? GitLab сам по себе не запускает jobs — для этого у него есть отдельный компонент **GitLab Runner**.

GitLab Runner — это агент, который подключается к GitLab и выполняет jobs из `.gitlab-ci.yml`. Когда мы пушим код, GitLab передаёт инструкции раннеру, и тот запускает всё, что нужно: команды, сборки, тесты, деплой и всё, что описано в нашем пайплайне.
Раннер может работать на разных типах инфраструктуры:
- на обычной виртуальной машине (например, в облаке);
- в Docker-контейнере;
- внутри Kubernetes-кластера;
- на реальном сервере внутри компании;
- на Linux, Windows и даже macOS.

Что именно делает раннер? Он берёт наш job, выполняет описанные в нём команды, собирает результаты и отправляет обратно в GitLab статус (успех или ошибка) плюс логи выполнения.

*Каждый job пайплайна исполняется изолированно, в своей среде, однако между вашими job'ами вы можете передавать артефакты и переменные, даже если они исполняются на разных раннерах*

К Gitlab можно подключить сколько угодно раннеров, на них можно вешать метки (например, по операционной системе), их можно делать выделенными для проекта или общими для всего Gitlab.
Для начала важно просто знать: `.gitlab-ci.yml` — это инструкция, которую выполняет агент в вашей или общей инфраструктуре.

## Stages и зависимости

### Логическое разбиение на stages

Мы уже говорили, что этапы (stages) задают общий порядок выполнения jobs в CI. Но важно понять, зачем вообще делить работу на этапы, а не просто писать список команд сверху вниз.
Ответ можно найти в философии DevOps. В ней жизненный цикл продукта — это не линейная прямая от идеи до релиза, а непрерывный цикл от планирования до мониторинга и обратно. Такой цикл часто рисуют в виде восьмёрки:

`Plan → Code → Build → Test → Release → Deploy → Operate → Monitor → (Plan)`

CI-пайплайн по сути реализует левую половину этой восьмёрки: от написания кода до релиза. Stages вроде `build`, `test`, `release` — это не случайный набор слов, а техническая реализация ключевых фаз разработки.
При этом результаты мониторинга и эксплуатации (Operate → Monitor) возвращаются в планирование и новые коммиты, замыкая цикл. Правильное проектирование пайплайна со stages, следующими этим шагам, не только ускоряет процесс, но и поддерживает культуру непрерывных улучшений и обратной связи DevOps.

*Не стоит дробить пайплайн на слишком много последовательных stages без необходимости. Каждый stage ждёт окончания предыдущего, и чем их больше, тем потенциально длиннее пайплайн. Логика разбиения должна отражать смысловые и технические зависимости.*

### Stages по умолчанию

В Gitlab CI по умолчанию есть набор stages: `.pre`, `build`, `test`, `deploy`, `.post`. Мы можем использовать его, а можем задать свои имена и свой порядок. Как мы говорили, все задания (jobs) внутри одного stage выполняются параллельно, а stage считается завершённым, только когда все запущенные jobs закончились. После этого стартует следующий stage.

Например:
```yaml
stages:
  - lint
  - build
  - test
  - deploy 
 ```

 Порядок `lint → build → test` типичен для компилируемых языков (Java, Go, C#), где тесты требуют собранных артефактов. В интерпретируемых языках (Python, Node.js без отдельной сборки) юнит-тесты иногда запускают до сборки — `lint → test → build`.

 Например:
```yaml
lint_backend:
  stage: lint
  script:
    - cd backend
    - pip install flake8
    - flake8 src/

lint_frontend:
  stage: lint
  script:
    - cd frontend
    - npm ci
    - npm run lint

build_backend:
  stage: build
  script:
    - cd backend
    - pip install -r requirements.txt

build_frontend:
  stage: build
  script:
    - cd frontend
    - npm ci
    - npm run build 
```

Здесь оба jobs линтинга запускаются параллельно первыми. Только после их завершения запустятся два jobs по сборке: `build_backend` и `build_frontend`. Они принадлежат одному этапу и тоже могут выполняться параллельно. Это упрощает проектирование, но иногда приводит к лишнему ожиданию. Например, если есть две независимые цепочки (бэкенд и фронтенд), они всё равно вынуждены дожидаться окончания всего предыдущего этапа.
Представьте: `build_backend` занимает 5 минут, а `build_frontend` — 30 секунд. Тесты фронтенда будут ждать 5 минут, хотя их зависимость готова уже через 30 секунд. Для таких случаев есть специальный инструмент `needs`.

### Needs — явные зависимости

Чтобы сделать пайплайн ещё быстрее, в GitLab CI есть директива `needs`. Она позволяет явно задать зависимости между jobs: тогда GitLab CI запустит job сразу после завершения тех jobs, от которых он зависит, не дожидаясь конца всего stage.
Пример:

```yaml
stages:
  - build
  - test
  - deploy

build_a:
  stage: build
  script: echo "Building A"

build_b:
  stage: build
  script: echo "Building B"

test_a:
  stage: test
  script: echo "Testing A"
  needs: [build_a]

test_b:
  stage: test
  script: echo "Testing B"
  needs: [build_b]

deploy_a:
  stage: deploy
  script: echo "Deploying A"
  needs: [test_a]

deploy_b:
  stage: deploy
  script: echo "Deploying B"
  needs: [test_b] 
```

В этом примере:
- `test_a` начнётся сразу после `build_a`, не дожидаясь `build_b`;
- аналогично, `deploy_a` стартует сразу после `test_a`, не ожидая завершения `test_b`.
Это сильно сокращает общее время пайплайна, если, например, `build_b` и `test_b` очень долгие, а `A` — быстрые.
Механизм `needs` ускоряет процесс, но требует аккуратности. Если все jobs переплести зависимостями, пайплайн станет запутанным. Трудно будет понять, где что сломалось, а визуализация в GitLab превратится в паутину.

*Особый случай — `needs: []` (пустой список). Это явное указание: «запусти этот job немедленно, не дожидаясь завершения ни одного предыдущего stage». Job стартует в самом начале пайплайна параллельно с первым stage. Это удобно для полностью независимых задач, но нарушает принцип Fail Fast: если такой job стартует до линтера, пайплайн продолжит работу даже если линтер упадёт.*

Иногда в следующем stage пайплайна нужно использовать результаты предыдущего. Например, это может быть собранный архив, бинарный файл или отчёт о тестах. В GitLab CI для этого есть механизм **артефактов (artifacts)** — они позволяют передавать файлы от одного job к другому. Если используется `needs`, GitLab сам позаботится о том, чтобы нужные данные были доступны:

```yaml
build_app:
  stage: build
  script: make build
  artifacts:
    paths: ["dist/"]

test_app:
  stage: test
  script: ./run-tests
  needs:
    - build_app 
```

Здесь test_app может сразу использовать результаты сборки (dist/), не дожидаясь других сборок, не ища файлы вручную и не скачивая что-то снаружи — GitLab автоматически скачивает артефакты из всех jobs, указанных в needs.
