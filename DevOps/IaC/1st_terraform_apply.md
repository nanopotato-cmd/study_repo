# Ваш первый terraform apply

## Язык разметки HCL

Желаемое состояние инфраструктуры декларативно описывается в файлах с расширением `.tf` с использованием языка разметки HCL (HashiCorp Configuration Language). Он похож на JSON, но более нагляден; его используют все инструменты Hashicorp. 

Манифест HCL состоит из блоков. Каждый блок описывает определённую сущность и её свойства. Например:
```hcl
тип_блока "вид_описываемой_сущности" "имя_сущности"{
 # решётка — это комментарий;
 # "имя_сущности" — опциональный параметр, иногда его нет.
 
 // кстати, два слеша — это тоже комментарий.
 
 # ниже описаны атрибуты блока
 # в отличие от JSON, HCL не требует фигурных скобок вокруг атрибутов и использует знак равенства вместо двоеточия.
 ключ1 = значение1 
 ключ2 = значение2
} 
```

Основные **верхнеуровневые** блоки в Terraform (в одном каталоге модуля):
- `terraform` — настройки самого Terraform: ограничения версии CLI и провайдеров, бэкенд для state, эксперименты. Это не описание «железа» в облаке.
- `provider` — как подключаться к API платформы (учётка, регион/зона по умолчанию, эндпоинты). Один провайдер может обслуживать много ресурсов.
- `resource` — создание и управление сущностью в инфраструктуре (ВМ, сеть, диск и т.д.); Terraform будет приводить реальное состояние к описанному в блоке.
- `data` — только чтение: ссылка на уже существующие объекты или внешние данные без обязательного управления ими из этого манифеста (часто — чтобы подставить ID в `resource`).
- `variable` — входные параметры модуля; значения задаются через CLI, файлы `.tfvars`, CI и т.д. Подробнее — в следующих уроках.
- `locals` — локальные вычисляемые значения внутри модуля: удобно собрать имя, список зон или повторяющееся выражение в одном месте, не создавая ресурс в облаке.
- `output` — что вернуть наружу после `apply`: IP, ID, DNS — для человека, для другого стека Terraform или для скриптов.
- `module` — подключение другого каталога с конфигурацией как кирпича: переиспользование и композиция. Разберём отдельно.

## Источники данных (`data`)

Блок `data` задаёт запрос к API провайдера на чтение. При `terraform plan` и `apply` Terraform получает актуальные поля объекта в облаке (например, `id`, `v4_cidr_blocks`, зону) и делает их доступными в выражениях. Запись в `state` для data не означает «мы владеем этим ресурсом». `terraform destroy` **не удалит** объект только потому, что вы его прочитали через источник данных.

Это нужно, когда подсеть, сеть или образ уже существуют (созданы в другом стеке, вручную или другой командой), а в текущем манифесте нужно подставить их идентификаторы в `resource`, не дублируя длинные UUID в каждом месте и не храня их «магическими строками».

Синтаксис тот же, что у `resource`: `data "<тип_источника>" "<локальное_имя>" { ... }`. На локальное имя ссылаются так: `data.<тип>.<имя>.<атрибут>`.

Например, можно взять уже существующую подсеть по её идентификатору и передать её `id` в сетевой интерфейс виртуальной машины:
```hcl
data "yandex_vpc_subnet" "existing" {
  subnet_id = "<идентификатор-подсети-в-облаке>"
}

resource "yandex_compute_instance" "vm" {
  # ... остальные аргументы ВМ ...

  network_interface {
    subnet_id = data.yandex_vpc_subnet.existing.id
  }
} 
```
Здесь `existing` — произвольное имя внутри конфигурации. По нему строится ссылка `data.yandex_vpc_subnet.existing.id`. В том же провайдере часто используют, например, `data "yandex_compute_image"` (получить `image_id` по семейству образов). В финальной практике спринта вы увидите такие же ссылки на результаты `data` — заранее имеет смысл запомнить форму `data.<тип>.<имя>.<атрибут>`.

Внутри блоков (например, у `resource "yandex_compute_instance" "vm"`) часто встречаются **вложенные блоки** — `boot_disk { ... }`, `network_interface { ... }`: это отдельная синтаксическая конструкция, не «присвоение через `=`».

Чтобы не запутаться, полезно помнить разницу между этими блоками.
`variable` — значение приходит извне (аргументы модуля, `.tfvars`),
`output` — значение уходит наружу после успешного применения. `locals` живёт только в конфигурации и state не «размножает» новыми облачными объектами. `data` не объявляет намерение создать ресурс — в отличие от `resource`, хотя в плане оба могут вызывать API для чтения.


## Провайдеры Terraform

Terraform создаёт инфраструктуру с помощью **провайдеров** — компонентов, которые отвечают за «связь» c API облака. Их можно воспринимать как «драйверы»: они преобразуют манифест в конкретные действия — создание, изменение или удаление ресурсов.

Провайдеры обычно предоставляются самой платформой или сообществом. Например, провайдер Яндекса позволяет взаимодействовать с Яндекс Облаком.

Процесс применения изменений можно разделить на несколько шагов:

1. **Определить желаемое состояние ресурса** — Terraform анализирует конфигурацию ресурса (`.tf`-файлы) и определяет, какими параметрами он должен обладать.
2. **Определить текущее состояние ресурса** — Terraform через провайдера получает информацию о текущем состоянии ресурса из облачной платформы или использует данные из файла состояния. Про него мы поговорим в следующих уроках. Пока важно понимать, что Terraform сохраняет сведения о созданных ресурсах в специальном служебном файле.
3. **Определить, нужны ли изменения** — Terraform сравнивает желаемое состояние ресурса (из манифеста) с текущим (полученным на предыдущем шаге) и определяет, требуется ли создание, изменение или удаление.
4. **Применить изменения** — если изменения действительно нужны, Terraform вызывает API облачной платформы через провайдер и выполняет нужные операции.

При работе Terraform придерживается принципа идемпотентности. То есть **повторный запуск не создаёт дубликаты сущностей, если они уже есть**. Утилита вносит только необходимые изменения, чтобы привести инфраструктуру к описанному состоянию.

Посмотрите файл ~/.terraformrc на ВМ:

```hcl
provider_installation {
  network_mirror {
    url = "https://terraform-mirror.yandexcloud.net/"
    include = ["registry.terraform.io/*/*"]
  }
  direct {
    exclude = ["registry.terraform.io/*/*"]
  }
} 
```

Разберём блок `provider_installation` из конфигурации Terraform построчно:
- `provider_installation` — начало блока конфигурации, определяющего, как Terraform должен устанавливать провайдеры;
- `network_mirror` — указывает, что провайдеры следует загружать из сетевого зеркала (локального репозитория), а не напрямую из официального реестра;
- `url = "https://terraform-mirror.yandexcloud.net/"` — адрес зеркала;
- `include = ["registry.terraform.io/*/*"]` — инструкция загружать из зеркала только те провайдеры, которые в оригинале размещены на `registry.terraform.io`.

Синтаксис `*/*` означает:
первая `*` — любое имя провайдера (например, `yandex`, `aws`);
вторая `*` — любая версия или подмодуль.

Иными словами, зеркало обслуживает все провайдеры с официального реестра. `direct {` определяет правила для прямого подключения к источникам провайдеров (минуя зеркало). Здесь мы задаём исключения.
`exclude = ["registry.terraform.io/*/*"]` запрещает Terraform напрямую обращаться к registry.terraform.io за провайдерами. Все запросы к официальному реестру будут перенаправлены на зеркало.

Cоздайте файл main.tf в рабочей директории, где вы укажете настройки провайдера:

```hcl
terraform {
  required_providers {
    yandex = {
      source  = "yandex-cloud/yandex"
      # Можно указать версию провайдера, с которой будет работать конфигурация.
      version = ">= 0.87.0"
    }
  }
  # Также можно указать версию CLI Terraform.
  required_version = ">= 0.13"
}

# Зона доступности (Availability Zone), где по умолчанию будут создаваться все ресурсы:
provider "yandex" {
  zone      = "ru-central1-a"
} 
```
Это важная часть конфигурации. Здесь мы указываем `required_providers` — Terraform понимает, что для работы нужно скачать провайдер `yandex-cloud/yandex`. Без него код не будет работать, потому что именно провайдер «переводит» описание ресурсов в запросы к API облака.

Отсюда следует, что версия провайдера имеет значение. В конфиге указано, что она должна быть не ниже `0.87.0`. 
Разные версии провайдеров могут заметно отличаться. Например, в версии `0.0.1` может не быть поддержки Kubernetes, потому что такая возможность появилась только в версии `0.0.5`. 

### Почему важно контролировать версию провайдера

Может показаться, что лучше всегда использовать последнюю версию провайдера. Но на практике это не всегда безопасно: новые версии могут содержать изменения, из-за которых часть вашего Terraform-кода перестанет работать. 
Особенно это касается мажорных обновлений (например, перехода с `1.x` на `2.x`) — в них часто встречаются несовместимые изменения.
Именно для этого в Terraform есть несколько способов указать версию провайдера (и модулей) в поле `version`. Они позволяют гибко задавать допустимые диапазоны. Разберём основные.

- **1. Базовые операторы сравнения**
Пример:
```hcl
version = ">= 0.87.0, < 0.90.0"  # от 0.87.0 до 0.89.x 
```

- **2. Тильда (~>) — «пессимистичный» диапазон**
```hcl
version = "~> 0.87.0" 
```
Разрешает обновления, не затрагивающие мажорную и минорную версию. Проще всего запомнить так: Terraform будет обновлять версию, но только в рамках той же ветки.
Например:
- `~> 0.87.0` — можно обновляться до `0.87.1`, `0.87.2`, но не до `0.88.0`;
- `~> 1.2.0` — можно обновляться до `1.2.x`, но не до `1.3.0`.

- **3. Комбинации условий**
Пример: `"~> 0.87.0, != 0.87.3"`

- **4. Специальные значения**
`""`(пустая строка) — разрешает любую версию (не рекомендуется для продакшена);
`"latest"` — использует последнюю доступную версию (рискованно: может сломать совместимость);
`"*"`, `"x"` — аналогично `""` (любая версия).

## Первые шаги запуска инфраструктуры

Настроив провайдера и получив токен для аутентификации, мы готовы разворачивать инфраструктуру с помощью Terraform. Как мы обсуждали выше, для описания используется язык разметки HCL в файлах расширения `*.tf`.
Чаще всего центральным местом описания инфраструктуры выступает файл `main.tf`, однако это не обязательное требование. Terraform читает все файлы `*.tf` и самостоятельно выстраивает зависимости манифестов.

Развернём виртуальную машину в Яндекс Облаке. Для этого мы опишем ресурс типа `yandex_compute_instance`. Внутри ресурса заполним основные параметры ВМ:
- `name` — имя машины.
- `resources` — количество ядер и оперативной памяти.
- `scheduling_policy` — политика размещения машины: в данном случае она прерываемая и может перезапускаться раз в сутки, что подходит для тестовых и некритичных приложений и экономит бюджет.
- `boot_disk` — загрузочный диск. Здесь указывается образ операционной системы.
- `network_interface` — идентификатор подсети, к которой будет подключена ВМ.
- `metadata` — SSH-ключ для доступа к ВМ.
- `allow_stopping_for_update` — параметр, который позволяет останавливать машину при изменении конфигурации — пригодится в дальнейшем.
- `platform_id` — какая аппаратная платформа.
- `secondary_disk` — отдельные тома под данные;

### Заполнили `main.tf`:

```hcl
terraform {
  required_providers {
    yandex = {
      source  = "yandex-cloud/yandex"
      # Можно указать версию провайдера, с которой будет работать конфигурация.
      version = ">= 0.87.0"
    }
  }
  # Также можно указать версию CLI Terraform.
  required_version = ">= 0.13"
}

# Зона доступности (Availability Zone), где по умолчанию будут создаваться все ресурсы:
provider "yandex" {
  zone      = "ru-central1-a"
} 

# Тип блока, тип ресурса, имя блока
resource "yandex_compute_instance" "vm-1" {
    name = "lesson-terraform-s16137092"
    resources {
        cores  = 2
        memory = 2
    }

    scheduling_policy {
        preemptible = true
    }

    boot_disk {
        initialize_params {
            image_id = "fd845dr9j4h2aaq1m6ko" # Поиск подходящего ID образа: yc compute image list --folder-id standard-images | grep ubuntu-22-04-lts
        }
    }

    network_interface {
        subnet_id = "e9ba6d6t1aggkangnqih" # Список подсетей: yc vpc subnets list 
        # Параметр nat включает присвоение публичного IP машине.
        nat       = false
    }

    metadata = {
        ssh-keys = "ubuntu:${file("~/.ssh/id_rsa.pub")}"
    }
    
    allow_stopping_for_update = true
}
```

### Работа с запущенной инфраструктурой и применение изменений

Как мы помним, Terraform — идемпотентный инструмент. Это значит, что при повторном запуске `terraform apply` или `terraform plan` он покажет, что изменений нет:
```sh
ubuntu@epdab3m2abgdhgqjm1go:~/terraform_module3$ terraform plan
yandex_compute_instance.vm-1: Refreshing state... [id=fhmo5gt56988u62edtpc]

No changes. Your infrastructure matches the configuration.

Terraform has compared your real infrastructure against your configuration and found no differences, so no changes are needed.
```

Измените параметр `nat` на значение `true` в файле `main.tf` и снова выполните `terraform apply`.

Символ `~` означает `update in-place` (обновление на месте).

```sh
ubuntu@epdab3m2abgdhgqjm1go:~/terraform_module3$ terraform plan
yandex_compute_instance.vm-1: Refreshing state... [id=fhmo5gt56988u62edtpc]

Terraform used the selected providers to generate the following execution plan. Resource actions are indicated with the following symbols:
  ~ update in-place

Terraform will perform the following actions:

  # yandex_compute_instance.vm-1 will be updated in-place
  ~ resource "yandex_compute_instance" "vm-1" {
        id                        = "fhmo5gt56988u62edtpc"
        name                      = "lesson-terraform-s16137092"
        # (16 unchanged attributes hidden)

      ~ network_interface {
          ~ nat                = false -> true
            # (10 unchanged attributes hidden)
        }

        # (5 unchanged blocks hidden)
    }
```

**Terraform умный**: он не пересоздаёт ресурс без необходимости. Если параметр можно изменить на лету (как `nat`), он делает **in-place update**. Только когда изменение требует пересоздания (например, смена зоны или образа), Terraform удалит старую ВМ и создаст новую.

Это одно из ключевых преимуществ IaC — минимальный даунтайм при изменениях.

# Работа с VM через Terraform

## Подготовка окружения

| Шаг | Команда | Что делает |
|-----|---------|-------------|
| 1 | `cd ~/terraform_module3` | Переход в директорию с Terraform-конфигами |
| 2 | `export YC_TOKEN=$(yc iam create-token)` | Получение и установка временного IAM-токена для аутентификации в Yandex Cloud |
| 3 | `terraform plan` | Показывает, какие ресурсы будут созданы/изменены/удалены (проверка без применения) |

## Создание/обновление инфраструктуры

| Шаг | Команда | Что делает |
|-----|---------|-------------|
| 4 | `terraform apply` | Применяет изменения: создаёт, обновляет или удаляет ресурсы согласно конфигурации |
| 5 | Ввести `yes` | Подтверждение операции (без этого изменения не применятся) |

## Удаление инфраструктуры

| Шаг | Команда | Что делает |
|-----|---------|-------------|
| 6 | `terraform destroy` | Планирует уничтожение всех ресурсов, описанных в конфигурации |
| 7 | Ввести `yes` | Подтверждение удаления (без этого ВМ и другие ресурсы останутся) |

## Проверка статуса

| Действие | Команда | Что делает |
|----------|---------|-------------|
| Статус ВМ | `yc compute instance list` | Показывает список ВМ, их статус (RUNNING/STOPPED) и IP-адреса |
| Подключение к ВМ | `ssh ubuntu@<публичный_IP>` | Подключение к ВМ по SSH (требуется ключ) |

## Схема работы

```text
[Подготовка] → [plan] → [apply/yes] → [Работа с ВМ] → [destroy/yes] → [Очистка]
```

## Важные замечания

- Токен живёт недолго — при ошибке аутентификации повторите шаг 2
- `terraform plan` безопасен — он только показывает, что изменится
- `terraform apply` и `destroy` без `yes` не выполняются — это защита от случайных действий
- Если ВМ не нужна — обязательно выполните `destroy`, чтобы не тратить ресурсы облака
