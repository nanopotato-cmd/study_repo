# Переменные и секреты: гибкая конфигурация

Инфраструктура почти всегда существует в нескольких вариантах: разработка, тест, продакшн и т. д. У окружений разные имена ресурсов, зоны, мощности ВМ, теги и проч.

В этом уроке мы вынесем изменяемые параметры в переменные и будем задавать их через `*.tfvars`, разберём типы, значения по умолчанию и валидацию, затронем секреты (`sensitive`, переменные окружения) и инструменты для читаемости — `locals` и `outputs`. Сначала немного теории о том, как организуют файлы в проекте, затем перейдём к параметризации и переменным.

## Типичное дерево каталога и склейка `.tf`

Чтобы понять, как живёт Terraform в проекте, вернёмся к типичному дереву каталога. Вот пример осмысленного разбиения по файлам (имена условны, главное — разделение по ролям):

```text
infra-dev/
├── main.tf                 # Подключаем провайдеров (Yandex) и описываем сами ресурсы (например, серверы)
├── networks.tf             # Описываем ресурсы (например, сети)
├── s3.tf                   # Описываем ресурсы (например, s3-бакеты)
├── variables.tf            # Блоки variable — входы конфигурации. Объявляем только то, что переменная существует, задаём её тип и описание без конкретных значений
├── outputs.tf              # Блоки output — что отдаём наружу. Описываем, какие данные мы хотим увидеть в консоли после успешного создания (например, IP сервера). Для секретных значений нужно установить sensitive = true
├── locals.tf               # Блок locals { }: позволяет не размножать длинные выражения по ресурсам. Это не входные переменные, а скорее env для кода. Позволяет вычислить сложные значения один раз, чтобы не дублировать длинный код в main.tf
├── terraform.tfvars        # Только присваивания вида имя_переменной = значение для уже объявленных variables.tf. Стандартное имя Terraform сам подхватит при plan/apply
├── dev.auto.tfvars         # Файл для конкретного стенда (dev). Суффикс auto заставляет Terraform подхватить его автоматически
├── .terraform/             # Сюда при init скачиваются тяжёлые бинарные файлы провайдеров и модулей. Можно добавить в .gitignore. Полезен для кеша локально и в CI. Скачает зафиксированные вами версии провайдеров и модулей
├── .terraform.lock.hcl     # Зафиксированные версии провайдеров обычно коммитят в Git. Гарантирует, что у коллеги скачается та же версия, что и у вас
├── terraform.tfstate       # С этим вы уже знакомы. Если объявлен бэкенд, они лежат удалённо
└── terraform.tfstate.backup 
```

*Все `.tf-`файлы в одной директории Terraform склеивает в единую конфигурацию. Порядок файлов не имеет значения — важны только имена блоков внутри и ссылки между ними.*

### От отдельных каталогов к одной конфигурации

Есть рабочая конфигурация для `dev`-окружения. Теперь нужен `prod`, завтра — `test`, послезавтра — ещё несколько стендов.
Инфраструктура компании динамична: при взрывном росте продаж нужно быстро «надуть» — добавить инстансы, в период затишья — «сдуть», чтобы не платить за простаивающие серверы. Это нужно закладывать в архитектуру заранее.

**Литерал** — это конкретное, жёстко заданное в коде значение (текст в кавычках, число или `true`/`false`), которое Terraform воспринимает как есть, а не как ссылку на переменную или функцию.

Дальше по этой логике придётся плодить `infra-stage/main.tf`, `infra-qa/main.tf`, `infra-da/main.tf` — менять в каждом литералы и рано или поздно окончательно запутаться.

Решение — **параметризация манифестов**. С помощью переменных задаются значения входных данных, которые влияют на результат. В итоге мы оперируем одним набором манифестов для всех окружений, меняя только набор переменных.

## Переменные

В HCL у каждой переменной есть **имя** (уникальное в пределах модуля) и **тип** (`type = ...`). Обращаться к значению в коде можно через `var.<имя>` (например, `var.subnets_list`). Переменные определяются в `.tf-файлах`, обычно в `variables.tf`. Это нужно для настройки инфраструктуры — так обеспечивается гибкость конфигурации, а также разделение кода и данных.

### Простые типы

Подойдут, когда значение одно:

| Тип в HCL | Примеры значений                  |
|-----------|-----------------------------------|
| string    | `"ru-central1-a"`, `"10.0.1.0/24"`|
| number    | `2`, `10.5`                       |
| bool      | `true`, `false`                   |

### Составные типы

1. **Список `list(<тип элемента>)` — упорядоченный** набор значений **одного** типа: есть нулевой элемент, первый, второй... Удобно для подсетей, списка зон и т. п.

```hcl
variable "subnets_list" {
  type    = list(string)
  default = ["10.0.1.0/24", "10.0.2.0/24"]
} 
```

2. **Словарь `map(<тип значения>)`** — набор пар **ключ → значение**, где все значения одного типа. По сути — это таблица соответствий, которую удобно использовать для выбора настроек в зависимости от ситуации.

Например, вместо множества `if/else` создаём карту, где ключи — `dev`, `stage`, `prod`, а значения — мощность серверов (`cores = 2, 4`). Ключ для вызова значения элемента берут в кавычки или квадратные скобки: `var.cpu_by_tier["dev"]`:

```hcl
variable "cpu_by_tier" {                              /* объявили переменную с именем cpu_by_tier */
  description = "This is a variable of type map"      /* описание переменной, также используется для генерации документации к модулю */
  type = map(number)                                  /* объявили тип переменной map с типом number */

  default = {                                         /* дефолтные значения, которые будут присвоены, если нигде в коде мы не зададим другие */
    dev  = 2
    prod = 4
  }
}                                                     /* закрыли объявление переменной */

resource "yandex_compute_instance" "vm" {             /* создаём ресурс с типом yandex_compute_instance с именем vm */
  name = "lesson-vm"                                  /* выше имя ресурса (как переменной), а здесь — имя ВМ, как она будет называться в облаке после создания */
  resources {
    cores  = var.cpu_by_tier["dev"]                   /* берём количество ядер из карты по ключу dev (то есть два ядра для dev) */
    memory = 4                                        /* задали объём оперативной памяти в ГБ */
  }
}                                                     /* закрыли создание ресурса */
```

3. **Объект `object({ ... })`** — несколько **именованных полей** с **разными** типами (похоже на структуру в языках программирования). От `map` отличается тем, что набор полей и их типы **жёстко заданы** в блоке `type = object({ ... })`. Это значит, что из простых кирпичиков (`string`, `number` и `bool`) можно собирать сложные конструкции, чтобы не плодить сто отдельных переменных, а передать их одной пачкой:

```hcl
variable "settings" {                   /* открываем блок объявления переменной с именем settings */
  type = object({                       /* определяем тип как object (строгая структура для сборки сложной конструкции всей переменной) */
    version   = number                  /* поле version: ждём только число (например, версию БД) */
    disk_size = number                  /* поле disk_size: тоже число (например, гигабайты диска) */
    backup    = bool                    /* поле backup: только логическое true/false (вкл/выкл) */
    zone      = string                  /* поле zone: только текстовая строка в кавычках */
  })                                    /* закрываем описание схемы нашего объекта */

  default = {                           /* начинаем блок значений, которые подставятся сами */
    version   = 14                      /* числовой литерал для версии */
    disk_size = 10                      /* числовой литерал для размера диска */
    backup    = true                    /* булев литерал (по умолчанию бэкап включён) */
    zone      = "ru-central1-a"         /* строковый литерал с названием зоны */
  }                                     /* закрываем наполнение объекта данными */
}                                       /* закрываем всё объявление переменной */
```

Пример обращений:

```hcl
variable "server_config" {
  type = object({
    name = string
    tags = map(string)
    enabled = bool
    network_settings = object({
      subnet_id = string
      security_groups = list(string)
    })
  })
  default = {
    name = "super-server"
    tags = {
      environment = "development"
      project = "super-project"
    }
    enabled = true
    network_settings = {
      subnet_id = "subnet-1"
      security_groups = ["sg-1", "sg-2"]
    }
  }
} 
```

Имя сервера: `var.server_config.name` - *Поле `name` находится на верхнем уровне объекта `server_config`, поэтому к нему обращаются напрямую через точку.*
Настройки сети: `var.server_config.network_settings` - *`network_settings` — это вложенный объект, и такое обращение возвращает весь блок сетевых параметров целиком.*
Список security-групп: `var.server_config.network_settings.security_groups` - *`security_groups` вложен в объект `network_settings`, поэтому путь включает оба уровня.*
Значение тега `project`: `var.server_config.tags["project"]` - *`tags` — это `map(string)`, поэтому конкретное значение берётся по ключу в квадратных скобках.*
Признак активации конфигурации: `var.server_config.enabled` - *`enabled` — булево поле верхнего уровня, доступ к нему строится напрямую от корневого объекта.*

4. **Кортеж `tuple([<тип1>, <тип2>, ...])` — упорядоченный** набор из **фиксированного числа** элементов: у первого элемента (`[0]`) один тип, у второго (`[1]`) — другой и т. д. Удобно группировать связанные, но разнотипные значения (строку, число, логический флаг) в одной переменной, не множа отдельные объявления и не подбирая каждый раз подходящую коллекцию.

От списка `list` кортеж отличается тем, что в списке все элементы одного типа, а длина задаётся фактическим значением. У кортежа и длина, и тип каждой позиции **заданы заранее** — нельзя передать дополнительный элемент или подставить строку туда, где ожидается число, не изменив объявление типа.

```hcl
variable "app_flags" {                                    /* объявили переменную-кортеж из трёх элементов разных типов */
  description = "Префикс сервиса, число реплик, метрики вкл/выкл"
  type        = tuple([string, number, bool])             /* позиция 0 — string, 1 — number, 2 — bool */
  default     = ["api", 2, true]                          /* значение по умолчанию; порядок и типы должны совпадать с tuple([...]) */
}                                                         /* закрыли объявление переменной */
```

Обратите внимание: в примере выше к элементам кортежа мы обращаемся по индексу, как у списка:

`var.app_flags[0]` = строка `"api"`;
`var.app_flags[1]` = число `2`;
`var.app_flags[2]` = `true`.

Как это используется в конфигурации ресурса:

```hcl
variable "vm_tuple" {                                     /* имя ВМ, ядра CPU, включён ли отдельный мониторинг */
  type    = tuple([string, number, bool])
  default = ["lesson-vm", 4, false]
}

resource "yandex_compute_instance" "vm" {               /* создаём ресурс */
  name = var.vm_tuple[0]                                /* имя из первой позиции кортежа */

  resources {
    cores  = var.vm_tuple[1]                            /* число ядер — из второй позиции */
    memory = 2
  }

  labels = {
    monitoring = var.vm_tuple[2] ? "on" : "off"         /* третья позиция — bool в выражении. И далее выбор значения по расчёту тернарного оператора. Он работает как компактная проверка if-else в одну строку: если условие слева от знака вопроса истинно, выбирается первый вариант, если ложно — второй. */
  }
}
```

5. **Набор** `set(<тип элемента>)` (иногда его ещё называют множеством) — это неупорядоченная совокупность различных значений одного типа, в которой **нет повторов** и **не зафиксирован порядок**. Такие значения полезны, когда нужно исключить дубликаты (например, в случае списка уникальных ID пользователей или конфигураций).

### Итог: `set` выбирайте, когда модель данных — множество уникальных однотипных значений; если нужны порядок, индексы или повторяющиеся элементы, используйте `list`. Для пар «ключ—значение» по-прежнему уместны `map` и `object`.

Помимо типов переменных, в Terraform есть разные механизмы работы с ними. 

## Способы передачи Input-переменных

Значение одной и той же переменной можно задать по-разному. Чаще всего используют такую лестницу от простого к удобному для команд и окружений:

1. **Значение в блоке `variable`** — атрибут `default`. Это базовый вариант: «если снаружи ничего не передали, возьми вот это».

2. Файлы `*.tfvars` — обычный HCL-файл в формате `имя_переменной = значение` без блока `variable`. Файл с именем `terraform.tfvars` (и рядом с ним `terraform.tfvars.json`, если используете JSON) Terraform подхватывает **сам** при `plan`/`apply`. Так удобно хранить типовые настройки репозитория. Для `test` и `prod` делают отдельные файлы и подключают их явно.

Типичная структура каталога:

```text
infra/
├── main.tf
├── variables.tf          /* только объявления variable "..." */
├── terraform.tfvars      /* значения по умолчанию для команды; загружаются автоматически */
└── environments/
    ├── test.tfvars       /* набор значений для теста */
    └── prod.tfvars       /* набор значений для прода */
```

И теперь мы можем запускать манифест с разными параметрами для `test` и `prod`:
```sh
terraform apply -var-file="environments/prod.tfvars"
terraform apply -var-file="environments/test.tfvars"
```

В примере кода выше ключ `-var-file` можно указать несколько раз: файлы применяются поочерёдно, и если одна и та же переменная встречается в нескольких файлах, победит значение из последнего переданного.

3. Аргумент `-var` в командной строке — разовая подстановка без файла (явное определение переменной при вызове CLI):
```sh
terraform apply -var="instance_zone=ru-central1-a"
```

4. Переменные окружения с префиксом `TF_VAR_` — имя после префикса должно совпадать с именем переменной в Terraform (для `instance_zone` — `TF_VAR_instance_zone`):
```sh
export TF_VAR_instance_zone="ru-central1-a"

terraform apply
```
Удобно подставлять секреты и значения из CI/CD, не записывая их в файлы.

5. **Интерактивный ввод** — если переменная объявлена без `default` и ни один из способов выше не задал значение, Terraform попросит ввести его в консоли перед `plan`/`apply`.

А что с приоритетом источников переменных? Если значение задано несколькими способами, действует **самый высокий** в списке. Вот приоритеты  (от высшего к низшему):

1. `-var` в командной строке.
2. `-var-file`.
3. `terraform.tfvars` и `terraform.tfvars.json` (автозагрузка).
4. Переменные окружения `TF_VAR_<имя>`.
5. `default` в блоке `variable`.


## Output-переменные

**Output-переменные** (блок `output "имя" { ... }`) похожи на возвращаемые значения функции в языках программирования: после `apply` Terraform может показать важные данные в консоли и сохранить в state. Их используют, чтобы не выуживать вручную IP и идентификаторы из длинных цепочек атрибутов, передавать значения в другой модуль или внешний инструмент — CI/CD, Ansible и т. д. Кроме того, по некоторым из них Terraform строит графы зависимостей.

Объявления удобно складывать в отдельный файл `outputs.tf` — имя может быть любым с суффиксом `.tf` (например, `my_output.tf` тоже допустимо).
Например, мы хотим получить информацию о внутреннем и публичном IP-адресах создаваемой ВМ (`vm-1`):
```hcl
output "internal_ip" {                                                                /* имя для terraform output и для модулей */
  description = "Внутренний IP-адрес виртуальной машины"                              /* текст для человека и для автодокументации */
  value       = yandex_compute_instance.vm-1.network_interface[0].ip_address
}
/* value ссылается на ресурс -> Terraform свяжет вывод с ВМ */

output "external_ip" {
  description = "Внешний IP-адрес для NAT (если включён)"
  value       = yandex_compute_instance.vm-1.network_interface[0].nat_ip_address
}
```

На этапе `terraform plan` для ещё не созданной инфраструктуры часть значений будет помечена как «станет известно после `apply»`:
```hcl
Changes to Outputs:
  + external_ip = (known after apply)
  + internal_ip = (known after apply)
```

А после успешного **terraform apply** в конце вывода команды появятся итоговые значения output.

Типичный пример — связка с Ansible: Terraform создаёт инфраструктуру, Ansible настраивает ПО на ней. Для этого Terraform может автоматически сгенерировать список созданных ВМ в формате, который Ansible сразу подхватит.

С помощью `terraform output` можно описать многострочный вывод через `heredoc <<EOT ... EOT` и подставить IP через `${ ... }`:
```hcl
output "ansible_inventory" {
  value = <<EOT
[servers]
${yandex_compute_instance.vm-1.network_interface[0].nat_ip_address}
EOT
}
```

Сохранить результат в файл:
```sh
terraform output -raw ansible_inventory > inventory.ini
```

Вы получите **артефакт** для следующего шага пайплайна, не копируя IP вручную, а также создадите динамический инвентарь для Ansible.


## Зависимости между ресурсами и связь с output

**Зависимости** задают порядок: что Terraform создаёт, обновляет и удаляет первым. Они бывают неявными (по ссылкам в выражениях) и явными (метааргумент `depends_on`). Блоки `output` тоже участвуют в графе: значение `output` зависит от ресурсов и выражений, перечисленных в его `value`.

### 1. Неявные зависимости

Если в аргументе ресурса фигурирует другой ресурс (например, `network_id = ...`), Terraform сам выводит: «сначала тот, на кого ссылаемся». Рассмотрим пример:
```hcl
resource "yandex_vpc_network" "main_network" {         /* создаём облачную сеть */
  name = "main-network"
}

resource "yandex_vpc_subnet" "main_subnet" {             /* создаём подсеть внутри этой сети */
  name           = "demo-subnet"
  network_id     = yandex_vpc_network.main_network.id   /* ссылка на id сети — это неявная зависимость подсети от сети */
  zone           = "ru-central1-a"
  v4_cidr_blocks = ["10.1.0.0/24"]
}
```

В этом примере ресурс подсети ссылается на идентификатор сети через `yandex_vpc_network.main_network.id` — это неявная зависимость: подсеть не начнёт создаваться раньше сети.

### 2. Явные зависимости — `depends_on`

Иногда связь по данным не выражена ссылкой, но порядок создания всё равно важен. Типичный случай: балансировщик и бэкенды. Для таких ситуаций используют `depends_on` — список ресурсов, которые должны существовать до текущего шага.
Пример: ВМ с приложением должна появиться после ВМ с БД, даже если в конфиге приложения нет прямой ссылки на `db_vm`:

```hcl
resource "yandex_compute_instance" "db_vm" {            /* первая ВМ с базой данных */
  name = "db-server"
  resources {
    cores  = 2                                          /* число vCPU — как в остальных примерах урока */
    memory = 2                                          /* ГБ ОЗУ */
  }
  boot_disk {                                           /* загрузочный диск */
    initialize_params {
      image_id = "fd8ivd2vbnl77jgdq69a"
    }
  }
  network_interface {
    subnet_id = "your-subnet-id"                          /* подставьте идентификатор подсети из облака */
    nat       = true
  }
}

resource "yandex_compute_instance" "web_vm" {            /* вторая ВМ с приложением, обращающимся к БД */
  name = "web-server"
  resources {
    cores  = 2
    memory = 2
  }
  boot_disk {
    initialize_params {
      image_id = "fd8ivd2vbnl77jgdq69a"
    }
  }
  network_interface {
    subnet_id = "your-subnet-id"
  }

  depends_on = [                                         /* вот наша явная зависимость, мы ждём, пока db_vm будет готова */
    yandex_compute_instance.db_vm
  ]
}
```

### Local-переменные

Когда это полезно: когда одно и то же выражение повторяется, когда нужно собрать string, object или list из нескольких источников, когда хочется отделить внутреннюю логику модуля от его публичных input / output.

**Local-переменные** задаются в одном или нескольких блоках `locals { ... }`. Это внутренние именованные значения модуля: собираются из `var`, атрибутов ресурсов и функций, а обращаться к ним можно через префикс `local`. — обратите внимание, что именно `local` в единственном числе (например, `local.instance_zone`).

Это полезно, когда одно и то же выражение повторяется, когда нужно собрать `string`, `object` или `list` из нескольких источников, когда хочется отделить наш модуль от его публичных input и output:

```hcl
locals {
  vm_name     = "server-${var.env}-${var.project}"      /* собрали имя из нескольких переменных в одну строку */
  common_tags = {                                       /* подготовили общий список меток для всех ресурсов */
    project = "devops-course"
    owner   = "student"
  }
  instance_zone = var.instance_zone                       /* local-переменная может ссылаться на input-переменную */
}

resource "yandex_compute_instance" "vm" {               /* создаём ресурс, используя наши локальные значения */
  name   = local.vm_name                                /* подставили готовую сложную строку */
  labels = local.common_tags                            /* применили сразу весь набор меток */
  zone   = local.instance_zone
  # передали зону по цепочке zone <- local.instance_zone <- var.instance_zone <-  input-переменная instance_zone
}
```

Локальные переменные удобны, когда одно и то же значение повторяется в нескольких местах или требует форматирования (например, собирается из атрибутов другого ресурса). Значения неизменяемы и вычисляются заново при каждом terraform plan и terraform apply.

Input-переменные — это параметры снаружи, `locals` — вспомогательная логика внутри. Не стоит делать `local` из того, что должен настраивать пользователь модуля, и заменять одно другим.

Граница модуля проходит по variable и output — это правила взаимодействия с ним:

- значение входит в модуль снаружи → `variable`;
- значение выходит из модуля наружу → `output`;
- значение нужно только внутри, границу не пересекает → `local`.

## Terraform и секреты

**Секреты** (пароли, ключи API, токены и другие чувствительные данные) остаются уязвимыми, если лежат в открытом репозитории или попадают в лог CI.

Есть пара приёмов вокруг Terraform:
- Переменные окружения `TF_VAR_...` и параметры CLI `-var`/`-var-file` — значения можно не вносить в код.
- `sensitive = true у variable` и при необходимости у `output` — Terraform будет маскировать значение в выводе `plan`/`apply`. Важно: это не шифрование state. Секрет может сохраниться в state — защищайте удалённый бэкенд правами доступа и политиками облака.

Теперь поговорим про внешние хранилища. Мы уже вскользь упоминали HashiCorp Vault в контексте хранения секретов. В масштабе компании, департамента, команды и множества стендов секреты всё чаще отдают специальным сервисам. Они могут гибко и безопасно работать с секретами, предоставляя RBAC и множественные интеграции, в том числе с огромным количеством инструментов DevOps. Таких сервисов много, но вот те, с которыми вы точно столкнётесь:

- Hashicorp Vault — экосистема HashiCorp;
- Облачные менеджеры секретов: AWS Secrets Manager, Azure Key Vault, Google Secret Manager;
- в Yandex Cloud — Yandex Lockbox.
