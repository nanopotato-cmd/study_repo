# Динамические конфигурации: управление множественными ресурсами

Напомню, что мы пришли к тому, что копировать блок resource в манифесте нужное количество раз не самое хорошее решение, которое приведёт к очевидным проблемам. Такой подход нарушает принцип DRY (Don’t Repeat Yourself, "не повторяй сам себя"), критически важный для инфраструктурного кода. Ведь плодя файлы с разными описаниями одинаковых и схожих ресурсов мы получаем:

1. **Риск рассинхронизации:** при изменении параметра легко пропустить одну из копий.
2. **Сложность ревью:** нужно проверять в коде, что все копии на всех окружениях идентичны.
3. **Проблемы с Git:** при откате изменений легко потерять консистентность, потому что будет изменяться не код целиком, как часть gitflow, а части разных файлов.

Решением, которое может помочь справиться с задачей - управляющие конструкции:

- условные операторы,
- циклы,
- динамические блоки.

## Условные операторы

Используется **тернарный оператор** — единственная «условная конструкция» в выражениях HCL.
Тернарная условная опера́ция (от лат. ternarius — "тройной") — это реализованная во многих языках программирования операция, возвращающая второй или третий операнд в зависимости от значения логического выражения, заданного первым операндом.

Сначала вычисляется условие (оно должно давать `bool`), затем берётся либо левая ветка (истина), либо правая (ложь). Оба результата должны быть **совместимы по типу** с контекстом, куда подставляется выражение.
```hcl
condition ? value_if_true : value_if_false 
```

**Пример из практики:** при создании ВМ удобно подставить публичный SSH-ключ из файла на машине разработчика, а если файла нет — не падать на этапе парсинга, а подставить заглушку (или другое согласованное с командой значение), чтобы явно увидеть проблему в метаданных:
```hcl
metadata = {
  ssh-keys = "ubuntu:${                                     /* Строка для cloud-init или метаданных: пользователь и ключ в одном поле */
    fileexists(pathexpand("~/.ssh/id_rsa.pub")) ?           /* Условие: существует ли файл ключа после раскрытия ~ */
    file(pathexpand("~/.ssh/id_rsa.pub")) :                 /* Если да — читаем содержимое ключа */
    "SSH_KEY_NOT_FOUND"                                     /* Если нет, то вывод что ключа нет */
  }"
}
```

Тернарный оператор хорошо решает задачи "включить / выключить" или «выбрать одно из двух», но масштабировать инфраструктуру им неудобно: десять одинаковых ВМ не опишешь без перечисления или копирования. Для повторяющихся ресурсов нужны **циклы**.

## Циклы

Как мы уже сказали во вступлении, циклы в Terraform позволяют создавать **несколько экземпляров** одного ресурса (или модуля), не размножая блоки `resource` вручную. Это и есть практическое продолжение идеи DRY: одно объявление - много объектов в state.

Ниже — три разных механизма: `count` и `for_each` привязаны к **ресурсам**, выражение `for` — к **значениям** (в `locals`, `output` и т. д.). Их важно не путать.

### Цикл `count`

`count` — самый прямой способ: Terraform создаёт ровно N копий ресурса и нумерует их целыми индексами `0 ... N ... N-1`. Внутри блока доступен псевдоатрибут `count.index` (хранит порядковый номер текущей итерации).
Типичный пример — три однотипные ВМ, имена которых отличаются только суффиксом:
```hcl
resource "yandex_compute_instance" "vm-1" {
  count = 3                                         /* Будут созданы три экземпляра с индексами 0, 1, 2 */
  name = "lesson-terraform-${count.index}"
  # ... остальная конфигурация ...
} 
```

Часто `count` связывают с **длиной списка** из переменной: число машин совпадает с числом элементов, а имя берётся по индексу.
**То есть через переменную.** Вместо «зашитой» тройки в count = 3 берут **список имён** (или других параметров) из `variable`. Тогда Terraform сам посчитает, сколько ВМ нужно: столько, сколько строк в списке.

Здесь мы объявляем массив строк vm_names, а затем идём по нему циклом count, создавая набор ВМ с именами из массива и своим числовым индексом:

```hcl
variable "vm_names" {
  type    = list(string)                      /* порядок элементов = порядок индексов count: 0, 1, 2, ... */
  default = ["vm-1", "vm-2", "vm-3"]
}

resource "yandex_compute_instance" "vm-1" {
  count = length(var.vm_names)                /* Сколько ВМ создать — ровно столько, сколько элементов в списке */
  name = var.vm_names[count.index]            /* У каждой копии свой count.index и имя совпадает с элементом списка с этим номером */
  # ... остальная конфигурация ...
} 
```

**Особенность поведения** `count` — привязка к позиции в списке. Индекс `0` всегда означает «первый элемент текущего `var.vm_names`». Если вставить новый элемент в начало или в середину, «смысл» каждого индекса сдвигается.

### Цикл `for_each`

`for_each` создаёт несколько похожих ресурсов из **карты (map)** или **набора (set)** строк.
В отличие от `count`, где вы работаете с индексами (0, 1, 2...), в `for_each` вы работаете с **ключами**, которые имеют осмысленные имена.
Задайте **уникальный префикс** — например, свой логин — и подставляйте его в шаблон имени вместе с `${each.key}`:

```hcl
variable "name_prefix" {
  description = "Уникальный префикс (логин), чтобы имена не пересекались с другими в общей папке"
  type        = string
}

variable "vm_names" {
  type    = list(string)
  default = ["front", "back"]
}

resource "yandex_compute_instance" "vm-1" {
  for_each = toset(var.vm_names)
  name     = "lesson-terraform-test-${var.name_prefix}-${each.key}"
  # ...остальная конфигурация...
}
```

Пример создания трёх пользователей:

```hcl
# Вместо того, чтобы писать три раза resource "yandex_iam_service_account" "sa-1"...

locals {
  users = {
    "alice" = { role = "viewer", env = "dev" }
    "bob"   = { role = "editor", env = "prod" }
    "carol" = { role = "admin",  env = "prod" }
  }
}

resource "yandex_iam_service_account" "users" {
  for_each = local.users
  name     = each.key           # alice, bob, carol
  # other parameters...
}
```

### Цикл `for` (выражение)

Выражение `for` в квадратных или фигурных скобках не создаёт ресурсы. Оно строит новое значение — `list` или `map` из существующей коллекции. Так удобно готовить данные для `output`, `locals`, аргументов ресурсов и т. д.

Синтаксис для списка:
```hcl
[for <элемент> in <коллекция> : <выражение>]
```
Опционально добавляют фильтр: ... if <условие>.

```hcl
output "even_numbers" {
  value = [for num in [1, 2, 3, 4] : num if num % 2 == 0]
}
```
Если обернуть выражение в фигурные скобки и использовать =>, получится map:

```hcl
output "upper" {
  value = { for name in ["dev", "ops"] : name => upper(name) }
}
```

- `["dev", "ops"]` — это `list(string)`, не `map`;
- `name => upper(name)` задаёт пары «ключ → значение»; ключом становится исходная строка, значением — `upper(name)`;
- итоговый тип — `map(string)`.


## Динамические блоки

Циклы — не единственный способ в Terraform сократить повторения и упростить конфигурацию. Для этого также используют механизм **динамической генерации блоков** - **вложенные** блоки одинаковой формы.

`dynamic` задаёт шаблон вложенного блока и источник элементов. Для каждого элемента коллекции Terraform сгенерирует отдельный вложенный блок с подставленными полями. Блок `dynamic` состоит из: цикла `for_each` (итерирует по коллекции) и `content` (описывает блок, который сгенерируется для каждого элемента):
```hcl
dynamic "<имя_вложенного_блока>" {
  for_each = <коллекция>
  content {
    /* параметры блока */
  }
}
```
Он особенно полезен, когда у ресурса есть множество однотипных вложенных блоков, отличающихся лишь значениями переменных. Например, такая конструкция:
```hcl
dynamic "labels" {
  for_each = {
    environment = "prod"
    role        = "web"
    team        = "data-team"
  }
  content {
    key   = labels.key
    value = labels.value
  }
} 
```
В данном примере три пары ключ-значение:
`environment = "prod"`
`role = "web"`
`team = "data-team"`

без использования динамической генерации описывалась бы так:

```hcl
labels {
  key   = "environment"
  value = "prod"
}
labels {
  key   = "role"
  value = "web"
}
labels {
  key   = "team"
  value = "data-team"
}
```

Генерацию повторяющихся вложенных параметров удобно оформлять через `dynamic`, 
однотипные ресурсы создавать через `count`, 
а ресурсы с уникальными параметрами — через `for_each`.

## Краткая шпаргалка

| Задача                                                              | Что использовать                                           |
|---------------------------------------------------------------------|------------------------------------------------------------|
| Ветвление значения в выражении                                      | Тернарный оператор `cond ? a : b`; при необходимости — `coalesce` / `try` |
| N однотипных ресурсов, порядок стабилен                             | `count`                                                    |
| Ресурсы по уникальным ключам, меняется состав                       | `for_each` (map или set)                                   |
| Построить list / map для `output`, `locals`, аргументов             | Выражение `for`                                            |
| Много одинаковых вложенных блоков внутри ресурса                    | `dynamic` + `for_each` + `content`                         |
