# Что такое IaC (Infrastructure as Code)

## 1. IaC: от хаоса к порядку

Представьте типичную компанию. Инфраструктура росла годами. Кто-то поднял сервер вручную. Потом другой инженер добавил балансировщик. Третий настроил базу данных. Через год никто не помнит, что где стоит. Если сервер упал — восстановить его за разумное время почти невозможно.

Это хаос. IaC превращает хаос в порядок.

**IaC** — это подход к управлению инфраструктурой, при котором конфигурация серверов, сетей, баз данных и других компонентов описывается в виде **машиночитаемых файлов (кода)**.
Эти файлы хранятся в Git, проходят code review, версионируются. Вы запускаете команду — и инфраструктура создаётся автоматически.

Буквально: код описывает, какие серверы нужны, какие сети, какие базы данных. Инструмент (например, Terraform) читает этот код и создаёт всё в облаке.


## Проблемы, которые решает IaC

| Проблема           | Без IaC                                | С IaC                                           |
|--------------------|----------------------------------------|-------------------------------------------------|
| Воспроизводимость  | "У Васи работало, у меня нет"          | Код всегда создаёт одинаковую среду             |
| История изменений  | Кто и когда поменял настройки?         | Git хранит полную историю                       |
| Drift (дрейф)      | Сервер живёт своей жизнью              | Код — единый источник истины                    |
| Код-ревью          | Нельзя проверить изменения             | Pull Request перед применением                  |
| Rollback           | Откат — это боль                       | Просто применить предыдущую версию              |
| Документация       | Устаревает мгновенно                   | Код сам себе документация                       |

## Подходы к построению IaC
### Переход от «серверов-снежинок» к «серверам-фениксам»

**Серверы-снежинки** уникальны и неповторимы, их нельзя точно воссоздать, если они исчезнут. Отсюда и название.

Когда инженеры часто и хаотично правят систему вручную, её уже нельзя собрать заново в том же виде. Так появляется дрейф конфигурации — ситуация, когда реальное состояние инфраструктуры постепенно расходится с изначально задуманной и утверждённой.

**Серверы-фениксы** — это инфраструктура, которая может «сгореть», а затем «воскреснуть» из кода. Подход IaC позволяет в любой момент поднять окружение заново в изначальном виде.

Инфраструктура, как код даёт возможность быстро создавать копии сред для тестирования и экспериментов. Все параметры инфраструктуры и приложения зафиксированы в коде, поэтому стенд можно воспроизвести даже через долгое время — без страха что-то сломать и без попыток вспомнить, кто и как настраивал оригинал.

Фениксы помогают быстро создавать копии окружений. Мы можем использовать манифесты, чтобы формировать тестовые среды даже спустя длительное время, так как вся информация есть в исходниках. Стенды можно настраивать, задавая параметры и конфигурации как для инфраструктуры, так и для приложения. Так можно получить окружение, максимально приближённое к оригинальному, и не бояться что-то сломать. В мире «серверов-снежинок» нам пришлось бы сначала вспоминать, кто и как поднимал исходный стенд, а потом отдельно разбираться, как масштабировать его под тестовые задачи.

### Подход Immutable Infrastructure

В офисе появился новый сотрудник, в первый же день он случайно изменил конфигурационный файл на продуктивном стенде и ушёл в пятничный закат. Поскольку его действия обошли привычные процессы, команде было трудно найти причину сбоя, что привело к длительному простою.

Чтобы не допускать таких ситуаций и остановить дрейф конфигураций (когда в систему вносят несанкционированные и незафиксированные правки), используют подход **Immutable Infrastructure**. Суть здесь проста: после развёртывания инфраструктурные компоненты не меняются. Вместо этого создаются новые версии, а старые удаляются.

Эта логика хорошо сочетается с идеей серверов-фениксов: не страшно потерять такой сервер, если через пару минут он поднимется заново в том же виде.

Например, в ситуации с обновлением ОС на множестве виртуальных машин, мы бы не обновляли каждую (даже с автоматизацией). Вместо этого построили бы систему так, что можно постепенно запустить новые ВМ с нужной версией ОС, а старые после прохождения тестов удалить.

## Два главных подхода

**Декларативный** (Terraform, CloudFormation). Вы говорите: "хочу три сервера вот с такими параметрами". Инструмент сам решает, что нужно создать, изменить или удалить.
*Декларативный подход* определяет конечное состояние ресурсов. Мы говорим инструменту, что нужно создать, — он сам разбивает задачу на шаги и выполняет их идемпотентно: повторный запуск ничего не меняет, если система уже в нужном состоянии.

**Императивный** (Ansible, Chef). Вы говорите: "сначала создай сеть, потом сервер, потом установи nginx". Пошаговая инструкция.
*Императивный подход* описывает последовательность команд для достижения цели. Идемпотентность здесь сложнее обеспечить, зато подход гибче.

В современном DevOps декларативный подход стал стандартом. Проще, надёжнее, легче поддерживать.

## Что даёт IaC на практике

**Воспроизводимость**. Вы можете развернуть идентичную среду в dev, stage, prod. Один код — везде одинаково.

**История**. Каждое изменение в Git. Всегда можно посмотреть, кто, когда и зачем поменял инфраструктуру.

**Безопасность**. Изменения проходят code review. Ошибка не попадёт в прод, не пройдя проверку коллегами.

**Скорость**. Вручную поднимать сервер — 20 минут. Через IaC — 30 секунд. И так 100 раз.

**Документация**. Код сам себя документирует. Никаких вики, которые устарели год назад.


## Основные концепции

**Желаемое состояние (desired state)**. Вы описываете, какой хотите видеть инфраструктуру. Инструмент сравнивает это с реальностью и приводит реальность к желаемому.

**Идемпотентность**. Запустили один раз — инфраструктура создалась. Запустили второй раз — ничего не изменилось, потому что уже есть. Запустили третий раз — опять ничего.

Повторный запуск `terraform apply` не меняет инфраструктуру, если она уже в желаемом состоянии.
```bash
terraform apply   # первый раз → создаёт сервер
terraform apply   # второй раз → видит, что сервер уже есть → ничего не делает
```

**Дрейф (drift)**. Кто-то вручную поменял настройки на сервере. Реальность разошлась с кодом. Инструмент это обнаружит и предложит вернуть как в коде.

## Инструменты

**Terraform** — бесспорный лидер для управления облачной инфраструктурой. Поддерживает AWS, GCP, Azure, Kubernetes, сотни других провайдеров.

**Ansible** — отличный выбор для настройки уже созданных серверов (установка пакетов, копирование конфигов). Но Ansible императивный, он выполняет шаги последовательно.

В идеале используют оба: **Terraform** создаёт инфраструктуру, **Ansible** настраивает ПО на ней.

## Главная мысль

IaC — не просто инструмент, это смена мышления. Инфраструктура становится кодом. Со всеми вытекающими: версионирование, ревью, CI/CD, тестирование.

## Практический пример на пальцах

### Без IaC (руками):

```bash
# 1. Зайти в AWS Console
# 2. Нажать "Launch Instance"
# 3. Выбрать AMI, тип, ключ
# 4. Создать Security Group
# 5. Настроить Network
# 6. Через час сделать то же самое для prod
# 7. А через месяц никто не помнит, как это делалось
```

### С IaC (Terraform):

```hcl
# main.tf
resource "aws_instance" "web" {
  ami           = "ami-0c55b159cbfafe1f0"
  instance_type = var.env == "prod" ? "t3.large" : "t2.micro"
}

resource "aws_security_group" "web" {
  ingress {
    from_port = 80
    to_port   = 80
    protocol  = "tcp"
    cidr_blocks = ["0.0.0.0/0"]
  }
}
```
```bash
terraform apply   # dev
terraform apply -var="env=prod"   # prod
```

Главное преимущество **Terraform** — умение скрывать сложность настройки инфраструктуры за декларативными описаниями. Вместо пошаговых инструкций мы задаём конечное состояние системы, а Terraform сам определяет, какие действия нужно выполнить. Это обеспечивает повторяемость, снижает влияние человеческого фактора и упрощает масштабирование.

Также Terraform умеет работать со множеством облачных провайдеров, поэтому может выступать как единый инструмент управления инфраструктурой (особенно для компаний с несколькими облаками). Он ведёт себя идемпотентно: повторное применение одной и той же конфигурации всегда даёт один и тот же результат.

## Сравнение подходов
Сравним два подхода на примере создания ВМ в Яндекс Облаке.
С `bash` и консольным клиентом `yc` команда выглядела бы, например, так:

```bash
yc compute instance create \
  --name "vm-name" \
  --folder-id "folder-id" \
  --zone "ru-central1-a" \
  --image-family "ubuntu-2204-lts" \
  --cores 2 \
  --memory 2 \
  --disk type=network-ssd,size=10 \
  --network-interface subnet-name=default-ru-central1-a,nat=true \
  --ssh-key "$(cat $HOME/.ssh/id_rsa.pub)" 
```

Сразу возникает вопрос: «А что, если нужно создать три таких ВМ, но в разных зонах доступности и с разными дисками?» Логика усложняется — придётся оборачивать команду в `bash`-скрипт с циклами и условиями.

Или: «Что, если сетевой диск должен создаваться только после успешного создания ВМ и сразу привязываться к ней?» Снова вручную прописывать зависимости в скриптах.

Без единого подхода скрипты быстро обрастают костылями — их становится тяжело писать, читать и поддерживать.

У такого подхода есть несколько важных нюансов:
1. На машине нужно иметь утилиту `yc` и все зависимости для работы `bash`.
2. Если нужен более сложный процесс создания инфраструктуры, команду придётся оборачивать в скрипты — и самостоятельно писать обращения к каждому эндпоинту API, обработку ошибок, ввод–вывод параметров, функции и их вызовы на `bash`.
3. Так как всё пишется на `bash`, нет единого стандарта описания команд: для большого проекта придётся вырабатывать единые правила оформления кода, тестирования и документации.

Если использовать специализированный IaC-инструмент (например, Terraform), та же команда выглядела бы так:

```hcl
provider "yandex" {
  token = "your_yandex_cloud_token"
  cloud_id = "your_cloud_id"
}

resource "yandex_compute_instance" "vm-name" {
  name = "vm-name"
  folder_id = "folder-id"
  zone = "ru-central1-a"
  resources {
    cores_count = 2
    memory_size_gib = 2
  }
  boot_disk {
    initialize_params {
      image_id = data.yandex_compute_image.ubuntu.id
    }
  }
  network_interface {
    subnet_id = data.yandex_vpc_subnet.default.id
    nat = true
  }
  metadata = {
    ssh-keys = "username:${file("~/.ssh/id_rsa.pub")}"
  }
}
```

Несмотря на более длинную запись, мы получаем единый синтаксис. Не нужно придумывать формат описания ресурсов, подходы к тестированию, масштабированию и изменению параметров: всё это уже заложено в Terraform и работает по стандартным правилам. Документировать инфраструктуру тоже проще — все ресурсы описаны в едином стиле. 

Когда проект растёт и требует масштабирования, командной работы и предсказуемости — IaC-инструменты становятся необходимостью. Переход на Terraform — это инвестиция: чуть больше времени на старте, чтобы сэкономить недели или месяцы в будущем.
