# Файл состояния и дрейф: контроль над реальностью

Код в репозитории задаёт желаемое состояние инфраструктуры, а в облаке в каждый момент существует фактическое. Если между ними нет надёжной связи и регулярной сверки, Terraform исходит из одних вводных, а в облаке уже существуют другие сущности. На практике это приводит к проблемам при `apply`, дубликатам ресурсов, нарушением зависимостей и риском повредить продакшн.

Связующее звено между манифестами и реальными ID объектов в облаке — **файл состояния (state)**. В этом уроке вы настроите **remote state** (хранение state в Object Storage) и разберётесь, зачем это нужно в командной работе и CI, научитесь находить и интерпретировать **дрейф (drift)** — расхождение между кодом и фактом в облаке, а также выполните импорт уже существующих ресурсов под управление Terraform без их пересоздания.

# Зачем нужен файл состояния?

В прошлом уроке мы говорили об идемпотентности Terraform — о том, что он не пересоздаёт сущности в облаке без необходимости, а только сверяет код с реальностью и вносит точечные изменения. Возникает логичный вопрос: откуда Terraform вообще знает, что уже существует в облаке?
Теоретически он мог бы каждый раз ходить в API облака и собирать статусы инстансов на лету, но на практике это медленно и неудобно. Поэтому Terraform хранит состояние в отдельном файле — `terraform.tfstate`. Это, пожалуй, самый важный компонент во всей работе с инструментом.

Файл состояния решает сразу несколько задач.

**Связка кода с реальными объектами.** В коде (`.tf`) вы пишете абстрактное имя (например, `resource "aws_instance" "my_server"`), но в облаке этот сервер получает уникальный ID (например, `i-0abc12345`). Terraform записывает эту связку в `state`, чтобы при следующем запуске понимать: `my_server` из кода — это конкретный инстанс в облаке.

**Хранение зависимостей.** Не все данные можно получить напрямую из облака. State хранит зависимости между ресурсами. Например, он помнит, что база данных должна создаваться строго после настройки сети — это помогает строить правильный граф ресурсов при запуске кода.

**Кеширование.** При большой инфраструктуре опрашивать API облачного провайдера по каждому ресурсу на каждый `plan` — долго. State работает как кеш и ускоряет работу.

Блокировка при совместной работе. Когда над проектом работают несколько инженеров или параллельно запускаются CI-пайплайны, state должен блокироваться. Если один человек выполняет `terraform apply`, файл блокируется — второй не сможет вносить изменения одновременно. Это предотвращает состояние гонки и порчу инфраструктуры. Механизм блокировки обеспечивается бэкендом Terraform, о котором мы подробнее поговорим ниже.

Если удалить `.tfstate`, Terraform потеряет всю инфраструктуру из виду. При следующем `plan` он решит, что ничего не создано, и попытается создать дубликаты всех ресурсов. Придётся либо удалять всё в облаке вручную, либо использовать `terraform import`, чтобы заново показать Terraform каждый существующий ресурс.

## Как выглядит state на практике

После успешного `terraform apply` с манифестом нашего `main.tf` в рабочей директории появляются два файла: `terraform.tfstate` и `terraform.tfstate.backup`. Оба относятся к одному и тому же состоянию, но выполняют разные роли.

- `terraform.tfstate` — основной файл состояния. Terraform хранит в нём связь между ресурсами в коде и объектами в облаке, их атрибуты и служебные данные. При `terraform plan` он сопоставляет код (`.tf`) с тем, что записано в state, и строит план: что создать, что изменить, что удалить. Без актуального *state* Terraform не знает, какие сущности уже «его».

**Два важных правила:** файл нельзя править вручную — риск рассинхронизации с реальной инфраструктурой слишком высок. Кроме того, в state нередко попадают чувствительные данные: пароли, ключи, IP-адреса. Храните его там, где ограничен доступ, и не коммитьте в открытый репозиторий без шифрования.

- `terraform.tfstate.backup` — резервная копия предыдущего состояния. При `apply` или `destroy` Terraform сначала сохраняет текущую версию в `.backup`, затем записывает новое состояние в `terraform.tfstate`. Это позволяет откатиться или сравнить версии, если что-то пошло не так.

Чтобы сравнить файлы можно c помощью утилиты `diff`:
```bash
diff terraform.tfstate terraform.tfstate.backup 
```

## Работа с состоянием Terraform

```bash
terraform state list # список текущих ресурсов, описанных в state
terraform state show yandex_compute_instance.vm-1 # атрибуты конкретного ресурса
terraform state --help # справка
```

`terraform state rm` удаляет запись о ресурсе из state, не обращаясь к API провайдера. Сам объект в облаке (ВМ, диск, сеть) при этом никуда не исчезает: Terraform просто перестаёт считать его частью управляемой инфраструктуры. Это не то же самое, что `terraform destroy` или `-target`, которые инициирует реальное удаление через провайдера.

Зачем это может понадобиться? Например, если в state попал лишний или ошибочно привязанный ресурс и нужно вывести его из-под управления Terraform, не трогая облако.

# Импорт в файл состояния Terraform

Допустим, вы выполняете `terraform plan` и видите, что данные в state не соответствуют реальному состоянию облака. Скорее всего, инфраструктура поднималась вручную, в обход IaC. Это частая ситуация — в реальности строго следовать парадигме «инфраструктура как код» удаётся не всегда.

Чтобы передать такой ресурс под управление Terraform, используют `terraform import`. Например, для ВМ, созданной вручную:
```bash
terraform import yandex_compute_instance.<Имя ресурса ВМ> <ID машины>
```
После того как инфраструктура переведена в код Terraform, любое ручное изменение через GUI облака снова создаёт дрейф конфигурации. Бороться с ним можно несколькими способами:

1. **Регулярные проверки через `terraform plan`** — в том числе джоба в CI, которая по расписанию проверяет инфраструктуру и алертит при неожиданных расхождениях.
2. **Запрет ручных изменений** — подход GitOps, когда все правки вносятся только через Git, а изменения в консоли облака запрещены.
3. **Immutable infrastructure** — ресурсы не изменяются, а пересоздаются.
4. **Блокировка state** — `state locking` предотвращает одновременные правки.
5. **Интеграция со сторонними сервисами** — например, Hashicorp Sentinel позволяет проверять код на соответствие политикам до применения изменений.

# Хранение состояния на удалённом ресурсе

Локальное хранение state рядом с манифестами создаёт несколько серьёзных проблем:
1. **Потеря файлов**. Несмотря на `terraform.tfstate.backup`, state легко потерять, просто удалив файлы.
2. **Безопасность**. Файлы состояния содержат конфиденциальные данные, поэтому их нельзя оставлять в открытом доступе вместе с основными манифестами.
3. **Совместная работа**. Когда над инфраструктурным кодом работают несколько человек, у каждого локально находится своя версия `terraform.tfstate` — из-за этого Terraform не может определить актуальное состояние инфраструктуры.
4. **Нет истории изменений**. Локально хранится только одна предыдущая версия — глубокой истории нет.

Решение — хранить state в удалённых ресурсах. Terraform предоставляет для этого множество бэкендов. По умолчанию используется бэкенд `local`, но есть и другие (например: `S3`, `pg`, `consul`, `http`). В рамках одного рабочего окружения можно использовать только один бэкенд.

Популярный вариант — **S3-бакет (Simple Storage Service)**, объектное хранилище для файлов. Здесь данные лежат в специальных контейнерах (бакетах). Взаимодействие с S3 происходит по `http`, а для доступа к бакету используется уникальный набор ключей. У Яндекс Облака также есть поддержка S3.

## S3-бакет

1. В разделе **Object storage**, выберите **Создать бакет** в правом верхнем углу.

Чтобы сервис-аккаунт смог получить доступ к бакету, ему необходим статический ключ доступа. Это не то же самое, что IAM-токен.

2. Найдите сервис-аккаунт `general-sa` с правами `editor` в разделе **Identity and Access Management**.
3. Выберите справа вверху **Создать новый ключ**, далее — **Создать статический ключ доступа**.

С помощью полученных ключей вы сможете обращаться к S3-бакету от имени сервис-аккаунта. Осталось настроить миграцию состояния из локального хранилища в S3.

```hcl
terraform {
  required_providers {
    yandex = {
      source = "yandex-cloud/yandex"
    }
  }

  backend "s3" {
    endpoints = {
      s3 = "https://storage.yandexcloud.net" # указать ендпоинты
  }
    bucket = "s16137092-terraform" # имя бакета
    region = "ru-central1"
    key    = "state/terraform.tfstate"

    skip_region_validation      = true
    skip_credentials_validation = true
    skip_requesting_account_id  = true
    skip_s3_checksum            = true
  }
}
```
4. После этого заполните переменные окружения полученными ключами:
```bash
export ACCESS_KEY="<идентификатор_ключа>"
export SECRET_KEY="<секретный_ключ>"
```

5. Инициализируйте Terraform
```bash
terraform init -backend-config="access_key=$ACCESS_KEY" -backend-config="secret_key=$SECRET_KEY"
```
