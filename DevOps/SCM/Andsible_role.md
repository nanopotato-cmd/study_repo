# Ansible-роли

Роли можно сравнить с библиотеками в программировании: библиотека решает конкретную задачу, и роль в Ansible делает то же самое — собирает задачи, шаблоны, файлы и переменные в один модуль для настройки компонента (например, веб-сервера или базы данных). Эти модули можно включать в плейбуки, не копируя постоянно код, а ссылаясь на роль в манифесте. Такой подход превращает описание инфраструктуры из набора скриптов в сборку из стандартизированных компонентов.

Основные проблемы, которые решают роли: дублирование кода, сложность поддержки и версионирование. Без ролей одинаковые наборы задач для настройки одного и того же ПО пришлось бы каждый раз копировать вручную, что неизбежно ведёт к ошибкам и разрастанию кода. Любое изменение общей логики потребовало бы править все такие копии вручную. Роли упрощают этот процесс: правка делается в одном месте и автоматически применяется везде, где роль используется. Кроме того, у ролей есть версии, что позволяет контролируемо обновлять зависимости в проектах.

Роли позволяют разбить большой запутанный манифест на набор маленьких и, что важно, независимых компонентов. Вместо одного монолитного плейбука, где выполняются все настройки окружения, итоговый манифест собирается из набора ролей — каждая выполняет свою задачу (например, установка базы данных или сервиса бэкенда). У всех ролей должны быть интерфейсы взаимодействия с ними (обычно это наборы переменных). Такой подход сильно упрощает управление инфраструктурой, так как инженер работает с модулем, не вникая в детали работы всей системы.

## Структура каталогов роли

Каждая роль должна иметь строго заданную структуру каталогов:

```text
role_name/             # Имя каталога, оно же имя роли
├── defaults/          # Переменные по умолчанию
├── vars/              # Переменные с высоким приоритетом
├── tasks/             # Задачи роли
├── handlers/          # Обработчики (перезапуск служб и т. д.)
├── templates/         # Файлы шаблонов
├── files/             # Статические файлы
├── meta/              # Метаинформация о роли
└── tests/             # Тесты 
```

Разберём подробнее каждую из директорий:

- Основная логика работы роли находится в директории `tasks/`, в файле `main.yml`, который служит точкой входа. В нём описываются задачи роли или подключаются другие файлы с задачами из этой же директории.
- Директории `defaults/` и `vars/` содержат переменные окружения, но имеют **разный приоритет**. Хорошая практика — хранить все значения по умолчанию в директории `defaults/`, а в `vars/` помещать константы, которые не должны переопределяться.
- Статические файлы (бинари, скрипты, конфиги) следует хранить в `files/`, а динамические шаблоны конфигураций, о которых речь пойдёт дальше, в `templates/`.
- Файл `meta/main.yml` хранит в себе не только метаданные роли (версии и описание), но и информацию о вызываемых зависимостях.
- `handlers` — директория с обработчиками. Это особый тип задач, которые могут выполняться при срабатывании определённых условий. Про них мы подробнее поговорим в следующем уроке.
- Сценарии тестирования роли (например, с помощью molecule) хранятся в директории `tests/`.


# Jinja2-шаблоны

Jinja2-шаблоны позволяют динамически генерировать конфиги на основе переменных и условий. Это даёт гибкость под разные окружения без правок самих шаблонов. Вместо множества статических конфигураций под каждый случай используют параметризованные шаблоны, где значения переменных подставляются в процессе работы плейбука. Это значительно сокращает количество ошибок, вызванных ручным редактированием конфигов, и обеспечивает единообразие настроек инфраструктуры.

Jinja2-шаблоны особенно нужны при управлении сложными распределёнными системами, где роли должны учитывать множество факторов: от специфики сетевых настроек и аппаратных характеристик до версий ПО и переменных окружения.

Генерацию манифестов из шаблонов выполняет модуль `template`. Код шаблонов заключается в нескольких конструкциях:
- Конструкция `{# #}` применяется для комментариев. Всё, что внутри, полностью игнорируется шаблонизатором и не попадает в итоговый файл.
- Конструкция `{{ }}` используется для вывода значений переменных или результатов выражений.
- Конструкция `{% %}` для управляющих операций (условные выражения и циклы). Она не выводит содержимое, а управляет ходом генерации файла из шаблона.

Шаблоны в Jinja2 имеют специальные функции, которые позволяют преобразовывать переменные, не прибегая к сложной логике в плейбуках. Это мощный инструмент, который превращает простую подстановку значений в механизм обработки данных. Вот несколько примеров базовых фильтров:

- `default` предоставляет значение по умолчанию, если переменная не определена (например, `{{ db_port | default(5432) }}`). Если переменная `db_port` пуста, то ей присвоится `5432`.
- `lower`/`upper` меняют регистр (например, `{{ app_name | upper }}`. Итоговое значение переменной `app_name` будет в верхнем регистре.
- `replace` выполняет простые текстовые замены `{{ path | replace(" ", "_") }}` (например, значение `hello world!` станет `hello_world!`).

Jinja2-шаблоны поддерживают условные конструкции внутри конфигураций, например:

```yaml
# Ключевая конструкция условия: {% if условие %}
{% if ansible_distribution == "Ubuntu" %}
Welcome to Ubuntu Server {{ ansible_distribution_version }}
# Если изначальное условие не выполнилось {% elif условие %}
{% elif ansible_distribution == "CentOS" %}
Welсome to CentOS {{ ansible_distribution_version }}
# Окончание блока с условиями
{% endif %} 
```

## Управление зависимостями ролей
В Ansible описывать зависимости ролей можно по-разному. 

Первый способ: описать зависимость через `meta/dependencies`. Это способ указать, что для работы необходимы зависимые компоненты и они должны быть установлены до начала выполнения текущей роли. Например, роль веб-приложения может зависеть от ролей базы данных и прокси-сервера. Этот способ **гарантирует**, что инфраструктура будет подготовлена в правильном порядке — СУБД, прокси, а затем само приложение.

Модули `import_role` и `include_role` нужны для **явной** организации зависимости ролей в плейбуке. Так вы контролируете порядок и условия запуска зависимых ролей и можете собирать инфраструктуру из независимых компонентов. Например: сначала развернуть базовую конфигурацию текущей роли, затем — систему мониторинга из наследуемой, и только после этого — основное приложение текущей роли. 

Ключевое отличие — в разных этапах выполнения. `import_role` **обрабатывается на этапе парсинга роли**, то есть её содержимое встраивается в основную роль. `include_role` же — **во время выполнения плейбука**, то есть её содержимое обрабатывается в момент вызова.

## Как писать роли

- **Продумать именования.** Можно следовать распространённому соглашению об именовании: `company.service-name`, это указывает на источник и назначение роли.
- **Сделать подробный `README.md`** с описанием роли, списком переменных и примерами её использования.
- **Большие задачи следует разбить на логические файлы**. Например: `install.yml`, `configure.yml`, `service.yml`, которые импортируются в `main.yml` с помощью модуля `include_tasks`.
- **Закрепить разницу** между `defaults/` (значения, которые пользователь может переопределять) и `vars/` (внутренние константы роли, не предназначенные для изменения). Учитывать иерархию переопределения переменных.
- **Правильно применять инструменты зависимостей:** `meta/dependencies` — когда зависимая роль должна изолироваться и выполняться до основной, `import_role` — для статических зависимостей, а `include_role` — когда нужно динамически - выбирать роль на основе условий или включать её в циклы.

# Продвинутые техники управления конфигурацией

В этом уроке мы познакомимся с продвинутыми техниками написания ролей и управления конфигурациями: вы узнаете, как работать с секретами в Ansible и обработчиками, чтобы повторяющаяся задача выполнялась только один раз, а также как управлять потоками выполнения плейбуков на множестве машин и обрабатывать ошибки.

## Ansible Vault

Когда мы используем Ansible в CI/CD-пайплайнах или при ручном запуске, часто возникает проблема передачи секретов (паролей, токенов, ключей). Решение, которое мы применили в прошлых уроках, — флаг `--extra-vars`для подстановки переменных напрямую в плейбук. Однако при масштабировании это приводит к катастрофе. 
Представьте пайплайн (например, в GitLab CI), где каждый вызов Ansible требует огромного количества переменных:
```yaml
script:  
  - ansible-playbook deploy.yml --extra-vars "db_password=$DB_PASS api_key=$API_KEY ssl_cert=$SSL_CERT ...и так далее..."
  ```

Чем больше секретов передаётся через `--extra-vars`, тем выше вероятность ошибок и тем сложнее их найти. Например, если переменную $API_KEY не экспортировали в CI-окружении, плейбук упадёт с неочевидной ошибкой. Поиск проблемы превращается в перебор десятков параметров. Кроме того, логи выполнения CI-заданий могут случайно отобразить секреты в открытом виде, что приведёт к дополнительным утечкам и уязвимостям.

Вот наглядный список проблем, которые могут возникнуть при использовании `--extra-vars`:
1. Громоздкая команда вызова плейбука при множестве переменных, сложности с его масштабированием.
2. Невозможность хранить плейбук и значения секретов в едином пространстве, так как есть риск утечки секретов.
3. Такой плейбук тяжело версионировать, так как секреты мы вынуждены хранить в отдельном месте.
Всё это решает **Ansible Vault**, позволяя безопасно управлять секретами в плейбуках. 

Он работает по принципу симметричного шифрования (когда для шифрования и расшифровки применяется один и тот же ключ), используя алгоритм AES-256 для защиты всего файла или отдельных переменных. Ключом является пароль, который может передаваться в Ansible через интерактивный запрос при выполнении или из файла.

## Обработчики событий (Handlers)

Они решают проблему избыточного выполнения задач, которые должны запускаться только при изменениях в системе.
Handler — это задача, которая выполняется только при изменении, вызванном другой задачей.

### Ключевые особенности

| Особенность         | Описание                                                       |
|---------------------|----------------------------------------------------------------|
| Выполняется один раз | Даже если вызван из нескольких задач                           |
| В конце плейбука    | Обработчики запускаются после всех задач                      |
| При изменении       | Только если задача действительно что-то изменила              |
| Идемпотентность     | Не перезапускает сервис без необходимости                     |

## Обработка ошибок плейбука

Вспомним, что в некоторых задачах используется опция `ignore_errors: yes`: она позволяет задаче завершиться с ошибкой и при этом не останавливает выполнение плейбука. Но что делать, если ошибку нужно обработать?
Для этого существуют блоки с секциями `rescue` и `always`. Они помогают сделать плейбук устойчивее: ошибка в задаче не обрывает выполнение целиком, а даёт возможность обработать сбой, выполнить откат или гарантированно запустить критические операции независимо от результата.

```yaml
- name: Обработка ошибок
  block:
    - name: Попытка установки пакета
      apt:
        name: some-package
        state: present
  
  rescue:
    - name: Если установка не удалась — использовать другой источник
      apt:
        name: some-package
        state: present
        deb: /tmp/some-package.deb
  
  always:
    - name: Выполняется всегда
      debug:
        msg: "Очистка временных файлов"
```

| Блок      | Когда выполняется                                       |
|-----------|---------------------------------------------------------|
| `block`   | Всегда (основная логика)                                |
| `rescue`  | Только при ошибке в `block`                             |
| `always`  | Всегда (и после `block`, и после `rescue`)              |

Блоки работают по тому же принципу, что и try–catch–finally в языках программирования. В секции block содержатся задачи, которые выполняются первыми. Если в них возникает ошибка, выполнение переходит в секцию rescue, где можно обработать сбой. Секция always выполняется в любом случае, независимо от результата выполнения блока.

## Управление потоком выполнения
