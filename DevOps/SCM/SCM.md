# SCM

**SCM (System Configuration Management)** — это инструменты для автоматического управления конфигурациями, настройки программного обеспечения и операционных систем. Они помогают поддерживать систему в стабильном и предсказуемом состоянии, предотвращая дрейф конфигурации.

Инструменты IaC и SCM тесно связаны: между ними есть различия, но SCM решает задачи, которые не всегда под силу IaC. Например, SCM управляет программным обеспечением на хостах, контролирует зависимости пакетов, гибко настраивает хосты и ПО, а также создаёт безопасное окружение для работы приложений.

Инструменты IaC работают в основном с API платформы (например, провайдер Terraform взаимодействует с API Яндекс Облака для запуска ресурсов). Инструменты SCM, напротив, чаще всего взаимодействуют с операционной системой физического или виртуального хоста. SCM тоже может работать с API, однако это не основной сценарий их использования.

Существуют два основных подхода к внесению изменений в систему: декларативный и императивный. 

- При **декларативном подходе** мы описываем желаемое состояние системы, а инструменты самостоятельно определяют, как его достичь.
- **Императивный подход** подразумевает, что мы описываем не конечный результат, а способ его достижения. Например, мы указываем SCM, как установить нужный пакет, и система выполняет наши инструкции.

Декларативный подход тесно связан с идемпотентностью: повторное применение конфигурации приводит систему к одному и тому же состоянию.

Подход (декларативный или императивный) не жёстко привязан к типу инструмента. Декларативный описывает желаемое состояние системы, императивный задаёт последовательность действий. На практике IaC-инструменты чаще используют декларативный подход (например, Terraform), SCM-инструменты могут быть как декларативными, так и императивными (Puppet и Chef соответственно).

Различие между IaC и SCM заключается не в подходе, а в области применения: IaC управляет инфраструктурными ресурсами через API, SCM — состоянием систем и приложений на хостах.

| Аспект               | IaC                                          | SCM                                          |
|----------------------|----------------------------------------------|----------------------------------------------|
| Объект управления    | Облачные ресурсы                             | Компоненты приложения                        |
| Метод описания       | Чаще декларативный (но возможны оба подхода) | Используются оба подхода (декларативный и императивный) |
| Метод работы         | Обычно разовое применение с сохранённым состоянием | Часто встречаются агенты, которые регулярно проверяют состояние системы; состояние системы часто не сохраняется |

Таким образом, получается очень эффективная связка: инструменты IaC создают инфраструктуру и подготавливают почву для инструментов SCM, которые настраивают её и доводят до готового состояния.

Основные модели взаимодействия — **push** и **pull** — определяют способ доставки изменений на хосты.

## Push- и Pull-модели

Инструменты SCM различаются способом доставки конфигурации на хосты. Существуют две основные модели работы:

- **push** — центральный сервер самостоятельно отправляет конфигурации на узлы;
- **pull** — агенты на хостах сами запрашивают конфигурацию с сервера управления.

Современные SCM-решения часто поддерживают обе модели одновременно — можно выбирать подходящий способ работы под конкретный сценарий.

В **pull-модели** на каждом хосте установлен агент, который периодически обращается к центральному серверу конфигураций и проверяет, соответствует ли текущее состояние системы ожидаемому. Если обнаруживаются расхождения, агент автоматически применяет необходимые изменения.
Такой подход тесно связан с идемпотентностью: повторный запуск агента не приводит к лишним изменениям, а лишь поддерживает систему в нужном состоянии.

В **push-модели** управляющий сервер сам подключается к хостам и запускает сценарии настройки — обычно по SSH, который есть практически на любой Linux-системе.
Push удобен там, где нужно быстро и синхронно применить изменения: обновить распределённое приложение или срочно раскатить патч безопасности. Слабое место — масштабируемость: при большой инфраструктуре управляющему серверу приходится одновременно держать множество соединений.

`Итого: push — для быстрых синхронных изменений, pull — для масштабируемых и отказоустойчивых инфраструктур`

Список хостов, которыми управляет Ansible, описывается в специальном файле под названием **inventory**. Он может быть в двух форматах: YAML или INI. В этом файле мы формируем списки серверов, указываем их адреса и параметры подключения. Мы будем использовать формат YAML, так как он более распространён.

Часто файлы inventory генерируются автоматически инструментами создания инфраструктуры, например с помощью `output` в Terraform. Информация о созданной инфраструктуре также может автоматически поступать от облачного провайдера. Ansible может получать эту информацию с помощью специализированных плагинов. Плагины обращаются в API облачного провайдера и получают список запущенных виртуальных машин. Это очень удобно в динамических средах и освобождает от необходимости постоянно вручную обновлять список ВМ в файле inventory при изменениях.

```yaml
all:

    practicum_servers:
      hosts:
        first_server:
          ansible_host: 111.88.241.159
        second_server:
          ansible_host: 89.169.181.147 
```

В выводе обратите внимание на `CHANGED` — модуль применил изменения. При повторном запуске статус сменится на `SUCCESS`: изменения уже внесены, делать ничего не нужно.

Каждая задача возвращает статус, отражающий результат её выполнения. Понимание этих статусов критически важно для анализа работы автоматизации:
- `SUCCESS` — система уже находится в нужном состоянии, Ansible не вносил изменений. Например, мы устанавливаем пакет, который уже есть на хосте.
- `CHANGED` — Ansible успешно применил изменения. Например, установил нужный пакет, которого не было на виртуальной машине.
- `FAILED` — ошибка выполнения: некорректный синтаксис, недоступный пакет, нет прав доступа. Плейбук останавливается, если не указано игнорирование ошибок.
- `SKIPPED` — задача пропущена из-за условий или ограничений. Например, настройка бэкенда не применяется к хостам фронтенда.
- `UNREACHABLE` — Ansible не смог установить SSH-подключение к хосту.

Модуль `raw` - Работет без установленного Python на удалённом хосте - Модуль raw передаёт команду напрямую.
Модуль `user` - Использует декларативный подход и проверяет текущее состояние - Модуль сверяет уже созданных пользователей перед применением.
Модуль `shell` - Выполняет команду без проверки состояния, но требует установленного Python - `shell` просто запускает команду.
Модуль `ping` - Проверяет доступность целевого хоста по SSH - При выполнении модуля ping, Аnsible устанавливает подключение по SSH и возвращает значение pong.

## Пишем первый плейбук

Выполнение однострочных команд очень удобно для решения небольших задач по автоматизации, но полная мощь Ansible раскрывается при написании плейбуков. 
**Плейбук (playbook)** — файл, описывающий наборы задач, которые будут выполнены на удалённых серверах.

Плейбук состоит из **плеев (play)** — секций, которые выполняются на указанных хостах, и **заданий (tasks)** — шагов, которые выполняют модули.

Например, небольшой плейбук для установки системы визуализации Grafana:
```yaml
# имя плея
- name: Install Grafana
# хосты из inventory или группа хостов, куда будут применены изменения 
  hosts: second_server 
# выполнять действия от имени суперпользователя
  become: yes 

  tasks:
    # задача установки Grafana
    # модуль: https://docs.ansible.com/ansible/latest/collections/ansible/builtin/apt_module.html
    - name: Install Grafana
      apt:
        update_cache: yes
        deb: https://dl.grafana.com/oss/release/grafana_12.1.0_amd64.deb

    # задача запуска демона 
    # модуль: https://docs.ansible.com/ansible/latest/collections/ansible/builtin/systemd_service_module.html
    - name: Enable and start Grafana service
      systemd_service:
        name: grafana-server
        state: started
        enabled: yes
```

Плейбук состоит из одного плея, который включает в себя две задачи: `Install Grafana` — установить пакет Grafana и `Enable and start Grafana service` — запуск демона.

```sh
ubuntu@fhmre7kljlnaejv9b89l:~/ansible-practicum$ ansible-playbook playbook.yaml
```

# Работа с переменными, фактами и условные конструкции

- научитесь использовать условную логику при написании плейбуков;
- узнаете, какие данные по умолчанию собирает Ansible c хостов;
- параметризируете свои плейбуки в зависимости от входных данных.

Это поможет вам создать универсальный плейбук, который адаптируется под различную инфраструктуру и учитывает все нюансы работы приложения.

## Переменные

Переменные в Ansible используются для придания гибкости и универсальности плейбукам. Они позволяют заменять «зашитые» значения (например, пути к файлам, имена пакетов, порты приложения) на динамические параметры. Это превращает статичный сценарий в шаблон, который **можно повторно использовать в различных окружениях** и для разных групп серверов без изменения самого кода. 

В Ansible одна и та же переменная может быть определена в нескольких местах: в `inventory`, в `group_vars`, в плейбуке, в файлах переменных и через CLI. Если это происходит, Ansible не объединяет значения, а выбирает одно — с наибольшим приоритетом.

Как понять, какое значение «победит»? Вместо заучивания списка удобно использовать простое правило: чем позже и ближе к запуску задаётся переменная — тем выше её приоритет.

| Источник                | Как воспринимать                                      |
|-------------------------|-------------------------------------------------------|
| `inventory`               | Базовая конфигурация инфраструктуры                   |
| `group_vars` / `host_vars`  | Уточнение для окружений и хостов                      |
| `vars` (в плейбуке)       | Значение внутри сценария                              |
| `vars_files`              | Явно подключённые данные                              |
| `--extra-vars`           | Ручное переопределение при запуске                    |


`vars_files` подключаются после `vars`, значит имеют более высокий приоритет. 
Каждый следующий уровень может переопределить предыдущий.
Общее правило: если переменная определена в нескольких местах внутри одного плейбука, используется значение, которое было подключено позже.

Итоговая логика такая:
- Сначала идут общие настройки (`inventory`);
- Потом настройки окружения (`group_vars`);
- Потом логика плейбука (`vars`);
- Потом явно подключённые данные (`vars_files`);
- В конце — ручное переопределение (`extra-vars`).

## Факты

Факты в Ansible — это встроенный механизм сбора детальной информации о хостах. При выполнении плейбука Ansible по умолчанию запускает специальный модуль `setup`, который собирает системные данные: от аппаратного обеспечения и дистрибутива операционной системы до сетевых интерфейсов и переменных среды. Эти сведения преобразуются в структурированную систему переменных, доступных для использования в плейбуке.

Сбор и вывод фактов группы серверов practicum_servers:
```sh
ansible practicum_servers -m setup                           # Все данные
ansible practicum_servers -m setup -a "filter=ansible_env"   # Фильтр по ansible_env
ansible <host> -m setup -a "filter=*keyword*"                # поиск по keyword
```

Понимание того, какие факты собирает Ansible, поможет сильно оптимизировать скорость выполнения плейбуков. Представим, что у нас есть 100 серверов и плейбук, который устанавливает только один пакет — зачем собирать информацию о дисках и виртуализации? Ansible может управлять этим процессом.

Факты группируются по категориям, и мы можем выбрать, какие собирать:

- `all` — все доступные факты (значение по умолчанию, но работает медленно на большом парке машин).
- `hardware` — используется, когда нужно адаптировать конфигурацию под железо (CPU, память, диски).
- `network` — для настройки сетевых сервисов и фаерволов.
- `virtual` — чтобы различать физические серверы и виртуальные машины.
- `!category` — исключить категорию из сбора.

## Условные конструкции

Условные конструкции — ключевой механизм, который делает сценарии гибкими. Они позволяют выполнять задачи или целые блоки задач выборочно, основываясь на результате условия. 
Условия могут проверять разные данные: значения переменных или фактов, результат выполнения предыдущих задач. Можно проверять не только простые равенства, но и существование файлов, вхождение строк в содержимое, сравнивать версии программного обеспечения или комбинировать несколько условий с помощью логических операторов.

| Описание | Условие |
|----------|---------|
| Запустить только на виртуальных машинах | `when: ansible_virtualization_type is defined and ansible_virtualization_type != "none"` |
| Пропустить задачу, если пакет уже установлен (код возврата проверки = 0) | `when: package_check.rc != 0` |
| Выполнить только на продакшен-серверах из группы production | `when: inventory_hostname in groups['production']` |
| Выполнить при неудачном результате предыдущей задачи | `when: previous_task is failed` |

## Регистры

Иногда в Ansible имеет смысл сохранить результат выполнения задачи во временную специальную переменную. Этот механизм называется регистром. С его помощью можно сохранять вывод команд, коды возврата и другие метаданные в переменные, которые затем становятся доступными для последующего использования. Это полезно в плейбуках, где дальнейшие действия зависят от результатов предыдущих операций — например, проверки существования файлов, статусов служб или валидации конфигураций.

```yaml
# Описание плейбука: установка и настройка Grafana
- name: Install and configure Grafana
  # Запускаем только на хосте first_server (из inventory)
  hosts: first_server
  # Выполняем задачи с повышенными привилегиями (sudo)
  become: yes

  # Блок переменных, доступных во всех задачах этого плейбука
  vars:
    # Версия Grafana, которую будем устанавливать
    grafana_version: 12.1.0

  # Список задач
  tasks:
    # Задача: проверить, установлена ли Grafana
    - name: Check if Grafana installed
      # Выполняем команду which (ищет исполняемый файл в PATH)
      command: which grafana-server
      # Сохраняем результат команды в переменную grafana_installed
      register: grafana_installed
      # Игнорируем ошибку, если команда завершилась с ненулевым кодом
      # (т.е. если Grafana не установлена — это ожидаемо)
      ignore_errors: yes

    # Задача: проверить доступность .deb файла по ссылке
    - name: Check Grafana deb link correct
      # Модуль для HTTP-запросов
      uri:
        # URL для скачивания .deb пакета
        url: https://dl.grafana.com/oss/release/grafana_{{ grafana_version }}_amd64.deb
        # Используем метод HEAD (не скачиваем тело, только заголовки)
        method: HEAD
        # Не возвращаем содержимое страницы
        return_content: false
      # Сохраняем результат (статус, заголовки) в переменную
      register: grafana_url_available
      # Игнорируем ошибки (если ссылка недоступна — проверяем статус позже)
      ignore_errors: yes

    # Задача: установить Grafana (блок из нескольких задач)
    - name: Installing Grafana for debian
      # Блок позволяет сгруппировать несколько задач
      block:
        # Подзадача 1: установить пакет Grafana
        - name: Install Grafana package
          # Модуль apt для управления пакетами Debian/Ubuntu
          apt:
            # Обновить кэш пакетов (аналог apt update)
            update_cache: yes
            # Установить .deb пакет по прямой ссылке
            deb: https://dl.grafana.com/oss/release/grafana_{{ grafana_version }}_amd64.deb

        # Подзадача 2: включить и запустить сервис
        - name: Enable and start Grafana service
          # Модуль для управления systemd-сервисами
          systemd_service:
            # Имя сервиса
            name: grafana-server
            # Состояние: запущен
            state: started
            # Автозапуск при загрузке системы
            enabled: yes

      # Условие выполнения всего блока (все 3 условия должны быть True)
      when:
        # 1. Grafana НЕ установлена (код возврата != 0)
        - grafana_installed.rc != 0
        # 2. ОС семейства Debian (Ubuntu, Debian и т.д.)
        - ansible_os_family == "Debian"
        # 3. Ссылка на .deb файл доступна (HTTP статус 200)
        - grafana_url_available.status == 200
```
